# DS 모델 Runtime 구동 메커니즘 — 모델링 룰 보강 자료

본 문서는 `1.entities.md` (entity 구조) / `2.modeling.md` (분해 룰) / `3.tooling.md` (도구) 를 **보강하는 보조 자료** 입니다. 모델링한 ds entity 가 *실제로 어떻게 실행되는지* 를 정리합니다 — 그래야 LLM 이 분해 판단 (예: capacity cycle vs Start-only, Self Reset 의 필요성, Call DAG 의 race-free 가정, ResetReset 의 의미) 의 *근거* 를 가지고 모델링할 수 있습니다.

> **본 문서의 권위 범위**: 보강 자료. `2.modeling.md` §3.7 (Arrow 시맨틱 / RGFH / capacity-2 cycle) 가 SSOT 이며, 본 문서는 그 SSOT 의 *근거 메커니즘* 을 자세히 풀어쓴 것. 모순 시 `2.modeling.md` 가 우선.

---

## §0 표기 약속 (본 문서 내)

`2.modeling.md` §3.7 의 DSL layer 표기를 그대로 사용:

- `-->` = Start arrow (solid — *순간 신호*: source 의 Finish rising edge)
- `..>` = Reset arrow (dotted — *지속 신호*: source 의 Going high level)
- `.->` = StartReset arrow (dash-dotted — Start ∪ Reset)
- `<..>` = ResetReset arrow (Passive 내부 opposing 쌍 / 양방향 reset)

자연어 layer 의 `→` 와 혼동하지 말 것. 사용자에게 결과를 보여줄 때는 *역할 어휘* 로 풀어쓰십시오 (`2.modeling.md` §0.2).

## §1 Runtime 모드 — 모델은 같지만 "구동 방식" 이 다르다

같은 ds 모델 (Project / DsSystem / Flow / Work / Call / ApiDef / Arrow) 이 4 가지 RuntimeMode 로 실행될 수 있고, 모드에 따라 **누가 상태를 결정하는가** 가 갈립니다.

| RuntimeMode | 누가 상태를 결정 | 엔진 구성 | 주 사용처 |
|---|---|---|---|
| **Simulation** | 엔진 자체 (시간/조건 시뮬레이션) | EventDriven (능동) | 모델 검증, 흐름 시연 |
| **Control** | 엔진 + 외부 PLC (양방향) | EventDriven (능동) | 실 PLC 와 동기 제어 |
| **Monitoring** | 외부 IO 신호 → 엔진이 *추론* | Passive (수동 관찰) | 실 라인 실시간 관찰 |
| **VirtualPlant** | 외부 가상 plant 신호 → 추론 | Passive (수동 관찰) | 가상 환경 PLC 학습 |

선택 기준:
```
RuntimeMode = VirtualPlant | Monitoring → Passive 엔진
RuntimeMode = Simulation | Control      → EventDriven 엔진
```

본 문서의 §3 ~ §11 은 **EventDriven (능동) 시맨틱** 입니다. Passive 모드의 추론 메커니즘은 §12 에서 분리해 다룹니다.

⚠ **모델링 의사결정 영향**: LLM 은 모델 *구조* 만 책임지면 됩니다. 모드 선택은 사용자/시스템이 결정. 단 "이 사양은 시뮬레이션용입니까 실 라인 모니터링용입니까?" 같은 질문이 *모델 구조* 를 바꾸는 경우는 없습니다 — 같은 entity 트리가 4 모드에서 모두 실행됩니다.

## §2 SimIndex — 모델을 실행 가능 형태로 컴파일한 결과

**정의**: ds 모델 (DsStore) 을 빌드 시 **1회** 계산하는 불변 메타 인덱스. 런타임 중 변경 불가.

**왜 LLM 이 알아야 하나?** SimIndex 가 분해 시 *암묵적으로 가정* 하는 그래프 속성이 몇 가지 있고 — 그 가정을 위반하는 모델은 런타임에 deadlock / 잘못된 token flow 를 일으킵니다. 본 문서 후속 절의 모든 시맨틱은 SimIndex 위에서 동작.

**SimIndex 의 핵심 맵 (요약)**:

| 맵 | 의미 | 모델링 영향 |
|---|---|---|
| `WorkStartPreds[w]` | w 의 Start 선행 Work 들 (Start / StartReset arrow source) | w 가 Going 으로 전이하려면 *모든* Start pred 가 Finish 도달해야 함 (AND) |
| `WorkResetPreds[w]` | w 의 Reset 지속 신호 source 들 (Reset / StartReset / ResetReset) | source 가 Going 인 동안 w 는 강제로 Homing → Ready (지속 신호) |
| `WorkTokenSuccessors[w]` | w 의 token-flow 후속 Work 들 | capacity cycle 의 token shift 경로 |
| `TokenSourceGuids` | token 발생점 (외부 신호 진입 Work) | Flow 의 *시작점* — 외부 트리거로 token 주입 |
| `TokenSinkGuids` | token 종료점 | token 이 sink 도달하면 자연 소멸 |
| `CallRaceExclusions[c]` | c 와 동시 Going 불가 Call set | **ArrowBetweenCalls (DAG)** 가 race-free 임을 보장하는 인덱스 |
| `WorkDuration[w]` | w 의 min duration (ms) | leaf Work 의 자체 진행 시간 (Call 없을 때) |
| ApiDef.TxGuid / RxGuid | Active Work 의 Call → Passive 내부 Work 의 신호 매핑 | helper 가 자동 채움 (룰 C / `2.modeling.md` §3.5) |

**Group arrow 의 자동 전개**: ds entity 가 reference (e.g. `ReferenceOf`) 를 통해 묶여 있으면 SimIndex 가 모든 멤버를 동일 그룹으로 등록 — 그룹의 한 멤버가 token 을 받으면 *모든 멤버가 동시 동기화*. (본 phase 도구는 `ReferenceOf` 를 직접 다루지 않으므로 모델링 시 명시적 사용 없음. Group arrowType 은 UI 클러스터링 hint — 실행 시맨틱 없음 / `2.modeling.md` §3.6.)

## §3 Work 의 RGFH FSM — 핵심 4 상태

```
       ┌──────── Start arrow source Finish↑ rising edge ────────┐
       │                                                          ▼
   ┌──────┐  duration 만료 + 모든 Call Finish     ┌────────┐
   │Ready │  ────────────────────────────────────▶│Going   │
   │  R   │                                       │  G     │
   └──────┘                                       └────────┘
       ▲                                              │
       │  Homing 정리 완료 (1ms+)                      │ duration met + all calls finish
       │                                              ▼
   ┌──────┐                                       ┌────────┐
   │Homing│ ◀────── Reset arrow source Going (지속)│ Finish │
   │  H   │   또는 token shift 후 자기 정리          │  F     │
   └──────┘                                       └────────┘
```

| 상태 | 의미 | 진입 조건 | 진출 조건 |
|---|---|---|---|
| **Ready (R)** | 시작 대기 | Homing 정리 완료 / 외부 reset | 모든 Start pred Finish + token 보유 + (선택) 외부 신호 |
| **Going (G)** | 실행 중 | Start 조건 충족 | min duration 경과 + 모든 하위 Call Finish |
| **Finish (F)** | 종료 직후 | Going 종료 | token shift 또는 Reset signal 도착 → Homing |
| **Homing (H)** | 정리 phase | Finish + Reset signal | 하위 Call 정리 + 1ms+ 경과 → Ready |

⚠ **single-instance 보장**: Going 상태인 Work 의 *새 Going* 은 시작 불가. 재실행은 반드시 F → H → R 사이클을 거쳐야 함. 이것이 capacity cycle 패턴에서 *Reset arrow 가 필수인 이유* — Reset 없이는 첫 사이클 후 영원히 Finish 에 머물러 다음 part 를 받지 못합니다.

⚠ **Homing 의 의미**: F→R 직접 전이가 아닌 이유 — Going 중 켜둔 device 신호 / 하위 Call 의 IO 출력을 *명시적으로 정리* 하는 phase. 하위 Call 들이 G→F 자동 스케줄되며, 최소 1ms duration 보장.

## §4 Call 의 R→G→F FSM — DAG race-free 보장

```
   ┌──────┐   Work going + ArrowBetweenCalls    ┌────────┐  IO 응답 (RX)  ┌────────┐
   │Ready │  ──── 선행 Call 모두 Finish ───────▶│Going   │ ──────────────▶│ Finish │
   │  R   │                                     │  G     │                │  F     │
   └──────┘                                     └────────┘                └────────┘
                                                  │
                                                  │ TX 신호 발행
                                                  ▼
                                              [Passive ApiDef binding]
                                              ApiDef.TxGuid Work → Going
```

**Call 은 Going / Homing 을 *외부에서 관찰하지 않음*** — 외부 시스템 호출이라 R 에서 ApiCall (TX) 발행 → 센서/응답 (RX) 도달 시 직접 F.

**ArrowBetweenCalls 의 race-free** (DAG 강제 / `2.modeling.md` §3.7):
- 같은 Work 안 두 Call 이 *동시 Going* 일 수 없음 (SimIndex.CallRaceExclusions 가 강제).
- DAG 위에서 *위상 정렬* 순으로 진행. 분기 (한 Call 이 두 Call 의 선행) 가능하지만 분기 후 두 형제 Call 이 *동시 Going* 일 수는 없음 — 순서는 SimIndex 결정.
- 한 Work 안에 사이클이 있으면 빌드 시 거부.

⚠ **모델링 영향**: 같은 station 의 sub-action (CLAMP × N → 메인 → UNCLAMP × N) 은 *같은 Work 안의 Call DAG* 로 묶어야 race-free 가 자동 보장됩니다 (`2.modeling.md` §3.4 / self-check 의 "한 station 의 모든 sub-action 같은 Work" 룰의 *근거*). 페이즈마다 Work 를 새로 만들면 Work-arrow (Cyclic) 영역으로 흘러가 race-free 가 깨질 수 있음.

## §5 Arrow 신호 시맨틱 — 순간 vs 지속

`2.modeling.md` §3.7 의 핵심 비대칭을 메커니즘 차원에서 풀어씀.

### 5.1 Start arrow (`-->`) — 순간 신호 (rising edge)

- source 가 Going → Finish 로 전이하는 *그 순간* 1 회 신호.
- target 이 Ready 이고 다른 Start pred (있다면) 도 모두 Finish 면 target Going 으로 전이.
- **edge-triggered** — source 가 Finish 에 *머물러 있어도* 신호는 1회만. target 이 그 사이 Ready 가 아니라 다른 상태 (예: Going) 였다면 신호 무시.
- 매 evaluation 에서 *모든 Ready Work* 의 Start pred AND 조건을 체크 → 충족 시 Going 전이.

### 5.2 Reset arrow (`..>`) — 지속 신호 (high level)

- source 가 Going 상태 *동안 계속* 신호 high.
- target 이 Finish 라면 → Homing 으로 강제 전이.
- target 이 다른 상태 (Ready / Going / Homing) 면 무시 — Reset 은 *Finish 만 깨움*.
- **level-triggered** — source 가 Going 인 *전 구간* 활성. source 가 Going 끝나면 신호 사라짐.

### 5.3 StartReset arrow (`.->`) — 두 신호의 union

- Start (rising edge) + Reset (high level) 양쪽 동시 활성. 즉 한 arrow 가 두 신호를 동시 송출.
- 의미: A `.->` B = (A 끝나면 B 시작) AND (A 진행 중이면 B 는 Finish 면 Homing 으로 강제 — 사실상 B 는 A 가 Going 인 동안 Ready/Homing 이외 상태에 머물 수 없음, 인터록).

### 5.4 ResetReset arrow (`<..>`) — Passive 내부 opposing 쌍 / 양방향 Reset

- A `<..>` B = (A.Going 동안 B Reset) AND (B.Going 동안 A Reset).
- 두 행위가 동시에 활성 불가능 — *상호 배타*.
- Passive 내부 opposing 쌍 (CYLINDER 의 ADV ↔ RET 등) 의 표준 사용처 (`2.modeling.md` §3.5).
- helper (`add_cylinder` / `add_clamp` / `add_device(opposing="chain")`) 가 자동 생성. LLM 이 직접 작성하지 않음.

### 5.5 Group arrow — 실행 시맨틱 *없음*

- UI 클러스터링 hint 만. 런타임 처리 없음.
- 단 SimIndex 는 `ReferenceOf` 기반 그룹 동기화 (UI Group arrow 가 아닌 reference) — 같은 그룹 멤버는 동일 token 상태 공유.

## §6 외부 신호 / 첫 Going 트리거

Flow 의 첫 Work (= TokenSource) 는 **선행 Start pred 가 없으므로** 자체 Start 조건이 없습니다. 다음 중 하나로 Going 트리거:

1. **외부 IO 신호** (HwButton, Hub 신호 — Passive/Control 모드).
2. **사용자 직접 트리거** (Simulation 모드의 Step / Start).
3. **Self Reset 으로 cycle 폐쇄** (capacity-1 반복 cycle) — Finish → 자기 자신 Reset → 즉시 Ready 복귀 후 다음 외부 신호 대기.

⚠ **모델링 영향**: Flow 의 TokenSource Work 는 *외부에서 token 을 받는 진입점* 이라는 의미. 사용자가 "라인 시작 버튼 / 도착 센서 / 알람 신호" 같은 트리거를 사양에 적었다면 → 본 phase 에서는 모델 구조에 포함하지 않으나 (HwButton 은 후속 phase) Flow 의 *첫 Work* 가 그 트리거의 진입점이라는 점을 사용자에게 안내.

## §7 Self Reset 패턴 — 자율 cycle 폐쇄

```
┌────────┐
│  W     │  ..> 자기 자신
│        │ ─────────────┐
└────────┘              │
     ▲                  │
     └──────────────────┘
```

- 의미: W 가 Going → Finish 로 전이하는 즉시 자기 자신을 Reset → Finish → Homing → Ready 1ms+ 만에 복귀.
- 효과: 외부 reset 신호 없이 W 가 자율적으로 cycle 폐쇄 → 다음 사이클 즉시 시작 대기.
- 표준 사용처: 반복 작업 Work 의 마지막 station — 다음 part 를 *바로* 받을 수 있게.

⚠ Self Reset 없이 단순히 외부 신호로만 다시 트리거하려면 사용자가 "매번 시작 신호를 다시 줘야 함" 을 *명시적으로* 의도한 경우 — capacity-1 의 1회성 처리. 사양에 "다음 부품 자동" 단서가 있으면 Self Reset 필요.

## §8 Capacity-N Cycle — 4-arrow 패턴의 동작 분해

`2.modeling.md` §3.7 의 capacity-2 cycle 패턴이 *왜 4 개 arrow 가 필요한가* 의 메커니즘:

```
trigger --> workA           (1) Start
workA --> workB             (2) Start
workB ..> workA             (3) Reset
workB ..> workB             (4) Self Reset
```

### 8.1 동작 step-by-step (한 part 단위)

| Tick | 외부 신호 | workA 상태 | workB 상태 | 발생 신호 |
|---|---|---|---|---|
| t0 | trigger 발생 | R → G | R | (1) Start |
| t1 | trigger 끊김 | G (Calls 진행 중) | R | — |
| t2 | workA Calls 완료 | G → F | R → G | (2) Start (A.Finish↑) |
| t3 | workB Going 시작 | F | G | (3) Reset (B.Going high) — A.F → H |
| t4 | A Homing 정리 (1ms+) | H → R | G | A 다음 사이클 받을 준비 완료 |
| t5 | B Calls 완료 | R | G → F | (4) Self Reset (B.Going→Finish↑ 자기 reset) |
| t6 | B Self-Homing | R | F → H | — |
| t7 | B Homing 정리 | R | H → R | 양쪽 다음 사이클 대기 |

### 8.2 t3 ~ t4 가 "동시 진행" 의 핵심

- t3 직전: 두 Work *동시 Going* 가능 (workA G + workB G). → **그래프 차원의 capacity = 2**.
- t3 에서 B.Going 의 Reset 신호가 A 를 Homing 으로 보냄 → A 가 다음 part 받을 준비.
- 즉 *workA 는 part1 처리 끝나자마자 part2 받을 수 있고, workB 는 그 사이 part1 을 처리* — pipeline.

### 8.3 N=3+ 확장

인접한 station 짝마다 동일 4-arrow 패턴 반복:
- A→B 짝: (A→B Start, B→A Reset, B→B Self)
- B→C 짝: (B→C Start, C→B Reset, C→C Self)
- → workA / workB / workC 동시 Going 가능 (capacity = 3).

⚠ **모델링 영향**: 사용자가 "동시 진행 / 다음 부품 바로 / 흐름 라인 / 파이프라인 / 처리량" 어휘를 쓰면 (`2.modeling.md` §3.4b) → 4-arrow 패턴 누락 시 첫 part 처리 후 라인이 막힙니다 (workA 가 Finish 에 영원히 머물러 다음 외부 신호 무시). self-check 의 "capacity 단서가 있는데 4-arrow 누락" 항목이 본 메커니즘 보호.

## §9 ResetReset (Passive 내부 opposing) — 디바이스 상호 배타의 메커니즘

CYLINDER 의 ADV ↔ RET, GATE 의 OPEN ↔ CLOSE 등 — 한 디바이스의 두 상반 행위가 동시 활성 불가.

```
PassiveSystem  Cyl
├─ Flow         Cyl_Flow
│   ├─ Work     ADV   (TxGuid=RxGuid 로 ApiDef ADV 와 binding)
│   └─ Work     RET   (TxGuid=RxGuid 로 ApiDef RET 와 binding)
├─ ApiDef       ADV
├─ ApiDef       RET
└─ Arrow        ADV <..> RET   ResetReset
```

### 9.1 동작 시맨틱

- Active 의 Call (e.g. `Cyl.ADV`) 이 Going 진입 → Passive 의 Work `ADV` 를 Going 으로 강제 (TxGuid binding) → device 신호 송출.
- 그 순간 ResetReset arrow 가 발화 → Work `RET` 가 만약 Finish 였다면 Homing → Ready 로 복원. 즉 *과거 RET 신호 흔적을 즉시 지움*.
- 반대 방향 동일 (RET Going → ADV Reset).

### 9.2 왜 도구가 자동 cascade 하는가

- helper (`add_cylinder` / `add_clamp` / `add_device(opposing="chain")`) 가 ResetReset arrow 까지 1 op 로 생성.
- primitive 만으로는 TxGuid/RxGuid binding + ResetReset arrowType 일관성을 손으로 맞춰야 함 — 실수 누적 시 "ADV 신호 줬는데 RET 도 동시 active" 같은 모델 깨짐.
- LLM 은 *helper 를 우선 사용* 하면 본 메커니즘이 자동 보호됨 (`2.modeling.md` §3.5 / `3.tooling.md` self-check 첫 항목의 *근거*).

### 9.3 단동 / 단일 action 디바이스는 ResetReset 없음

- 단동 실린더 / Drill / Spray / ROBOT — 상반 행위 자체가 없으므로 ResetReset 미생성 (`2.modeling.md` §3.3a).
- helper 호출 시 `opposing="none"` 명시 (default 값).

## §10 Token Flow — Group sync / Sink 도달 / Blocked

Work 사이를 흐르는 *논리적 토큰* (recipe / 부품 / 처리 단위) 의 메커니즘. 본 phase 도구는 TokenSpec / TokenValue 를 직접 만들지 않지만 (후속 phase) — 그래프 구조 (`WorkTokenSuccessors` / `TokenSourceGuids` / `TokenSinkGuids`) 자체는 **Arrow 만으로 결정** 되므로 모델링 시 영향 있음.

### 10.1 Token shift

- Work W 가 Finish 도달 → token 보유 중이면 후속 Work (W 의 Start arrow 후속) 들 검사:
  - 후속이 Ready 면 token 이동 → 후속 Going 트리거.
  - 후속이 Ready 가 아니면 → W 는 *Blocked* 상태로 머묾 (token 못 넘김).
- Sink Work (TokenSinkGuids) 면 token 자연 소멸.

### 10.2 Group sync

- `ReferenceOf` 로 묶인 Work 그룹은 *모든 멤버가 Going / Finish 상태 동기화*.
- 한 멤버라도 Going 못 가면 그룹 전체가 대기 — **deadlock 노출 설계**. 의도적 설계 점검 (사용자가 멤버 그래프를 잘못 그렸으면 deadlock 으로 알림).

### 10.3 Blocked 상태

- Work 가 Finish 인데 token 못 넘기는 상태 (후속이 모두 Ready 아님).
- Reset signal 도달 또는 후속 Ready 변화로 unblock.

⚠ **모델링 영향**: capacity cycle 의 Reset arrow (4-arrow 패턴의 (3)) 가 누락되면 — 첫 사이클 후 workA 가 Finish 에 머물러 token 을 다음 part 가 도착해도 못 넘김 (Blocked). 본 메커니즘이 self-check 의 "capacity 단서 + 4-arrow" 룰의 근거.

## §11 Duration / Min Duration — 시간 진행

- **Work.Duration** (entity 속성, ms): leaf Work (Call 없음) 의 자체 진행 시간. 사용자/도구가 명시.
- **Min Duration**: Work 가 Going 에서 Finish 로 전이하려면 (a) Duration 경과 AND (b) 모든 하위 Call Finish — *둘 다* 충족 필요.
- **Pause 중 Duration**: leaf Work 는 paused 시간도 carry over.
- helper (`add_cylinder` / `add_clamp`) 의 default Work duration = 500ms (Passive 내부 Work).

⚠ 본 phase 에서 LLM 이 Duration 을 자유롭게 설정하지 않음. helper default 그대로 두거나 사용자 명시 시만 반영.

## §12 Passive 모드 (Monitoring / VirtualPlant) — 추론 메커니즘

EventDriven 모드와 달리 **외부 IO 신호 → 엔진이 Work/Call 상태를 *추론***. 모델 구조는 동일하지만 동작은:

| 항목 | EventDriven | Passive |
|---|---|---|
| Work 상태 결정 | Scheduler 이벤트 (조건 평가) | 외부 IO 신호 변화 (관찰) |
| Duration | 시뮬레이션 타이머 | 외부 신호 간격으로 *측정* |
| Homing phase | 자체 정리 (1ms+) | 신호 사라짐으로 추론 — *명시 H 상태 없음* |
| Call 실행 | TX 신호 송출 | 수신만 (관찰) |
| Cycle 학습 | — | cycle 시간 / 흐름 학습 |

**Bootstrap**: 시작 시 모든 device 의 현재 IO 상태 snapshot → 초기 Work 상태 매핑. 그 후 신호 변화마다 다음 Work 상태 추정.

⚠ **모델링 영향 없음**: Passive 모드의 *추론 정확도* 는 모델 구조의 *정합성* 에 비례 — Active/Passive 분리, ApiDef.TxGuid/RxGuid binding, ResetReset opposing 쌍이 정확해야 추론 가능. 즉 §1~§11 의 모델링 룰이 EventDriven 뿐 아니라 Passive 모드의 정상 동작에도 필수.

## §13 모델링 의사결정 영향 정리 — LLM 가이드

본 문서의 메커니즘이 분해 결정에 미치는 영향을 정리:

| 모델링 결정 | 본 문서 근거 | 위반 시 런타임 결과 |
|---|---|---|
| **한 station 의 sub-action 은 같은 Work 안 Call DAG** | §4 (DAG race-free) | 페이즈마다 Work 분리 → Work-arrow Cyclic 영역 → race condition / token shift 오류 |
| **capacity 단서 시 4-arrow 패턴** | §8 (Reset 누락 시 Blocked) | 첫 part 후 workA 영원히 Finish — 라인 정지 |
| **Self Reset (반복 cycle 의 마지막 Work)** | §7 (자율 cycle 폐쇄) | 매번 외부 trigger 필요 — 사용자 의도 ("자동 반복") 위반 |
| **Passive 내부 ResetReset (opposing 쌍)** | §9 (helper 자동 cascade) | ADV/RET 동시 active 가능 — 실 device 충돌 |
| **ApiDef 는 Passive 안 (룰 C)** | §9 (TxGuid/RxGuid binding) | Active 의 Call 이 ApiDef 못 찾음 — `add_call` 에서 BATCH_ERROR |
| **Flow 경계 = 독립 트리거 흐름 (룰 E)** | §6 (TokenSource = 외부 진입) | 두 Flow cross-arrow → token flow 모호 / Flow 의미 상실 |
| **§3.3a 단동/복동 추론** | §9.3 (단동은 ResetReset 없음) | 단동인데 ResetReset 추가 → 복원 신호 강제 — 의도 위반 |
