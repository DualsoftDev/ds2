# 모델링 도메인 룰 — 사양 → Ds2 모델 분해

이 문서는 사용자가 자연어 사양 (예: "Cyl 을 전진 후 후진", "S301 에 실린더 2 개와 컨베이어 1 개", "로봇 2 대를 home → work1 → work2 순서로") 을 줄 때 그것을 **올바른 Project / DsSystem(Active|Passive) / Flow / Work / Call / ApiDef / Arrow 묶음** 으로 분해하는 의사결정 룰입니다. 3.tooling.md 가 "도구를 어떻게 호출하는가" 라면 이 문서는 "무엇을 만들어야 하는가" 입니다.

---

## §0 해석 단계 — 자연어 → Ds2 entity mapping

**전제**: 사용자는 Ds2 의 entity 어휘 (Project / DsSystem / Flow / Work / Call / ApiDef / Arrow) 를 *모를 가능성이 높습니다*. 사용자는 "공정", "라인", "스테이션", "디바이스 동작" 같은 *현장 어휘* 만 사용. 이 비대칭을 무시하고 사용자 어휘를 entity 에 *직접 1:1 매핑* 하면 길을 잃습니다 (예: 한국어 "공정" 을 자동 Flow 로 매핑 → §3.4a 의 (1) 케이스를 (2) 로 오인 → 룰 E 위반).

### 0.1 mapping 절차 (LLM 내부 수행)

1. 사양에서 *명사* / *동사* / *흐름 단서* 추출:
   - 명사 — 디바이스 인스턴스 (실린더, 클램프, 로봇, gun, spray 등), 라인 / 공정 / 스테이션 명, 컨트롤러 명.
   - 동사 — CLAMP / OPEN / SPOT / SPRAY 등 디바이스 동작.
   - 흐름 단서 — "후 / 다음 / then / after", "넘기면 / 이동", "동시 / 병렬", "다음 부품 바로", "한 station 에서 다음으로", "반복".
2. 각 명사 / 동사에 §1 매핑표 적용 → 잠정적 Ds2 entity 결정. 모호한 어휘 ("공정" 등) 는 §3.4a 결정 트리로 분기.
3. 흐름 단서에서 §3.4b / §3.6 / §3.7 의 arrow 패턴 결정 — capacity cycle vs Start-only vs ResetReset 등.
4. 절대 룰 (§2 A / B / C / D / E) 위반 검사. 위반 시 분해 단위 수정.
5. self-check (§5) 통과 후 `apply_model_doc` 1회 발행 (schema v0 JSON object).

### 0.2 사용자와의 대화 — *역할 어휘로 풀어쓰기*

명확화 질문 / 결과 요약 시 **DS entity 어휘 (Flow / Work / ArrowBetweenCalls / ApiDef / ResetReset Arrow 등) 를 직접 노출하지 마십시오** — 사용자는 그 어휘의 의미를 모를 가능성이 큽니다. 대신 *역할 어휘* 로 풀어 설명:

| 지양 — DS 어휘 직접 노출 | 권장 — 역할 어휘로 풀어쓰기 |
|---|---|
| "Flow `Process1` 안의 Work 3 개" | "공정 1 안의 station 3 개" |
| "ArrowBetweenWorks Start" | "한 station 끝나면 다음 station 시작" |
| "ResetReset Arrow" | "두 동작 중 하나만 활성 (상호 배타)" |
| "Passive System 의 ApiDef" | "디바이스가 제공하는 동작 (CLAMP / OPEN 등)" |
| "Work 안의 Call DAG" | "한 station 안의 sub-action 들 (순차 처리)" |
| "SelfClear 패턴 (dummy work + 2 arrow)" | "한 단계 끝나는 즉시 자기 자신을 ready 로 복귀 (다음 부품 받을 준비)" |
| "순차 work cluster 의 표준 반복 cycle 패턴" | "두 station 동시 진행으로 흐름 라인 구성" |

이름 / 식별자 (Project 명, System 명, Flow 명, Work 명) 는 사용자가 *원하는 이름* 그대로 보존 — 단 *entity 종류 어휘* 만 풀어쓰기. 사용자가 "공정 명을 무엇으로 할까요?" 같이 물어오면 그대로 받아 적용.

### 0.3 결과 확인 형식 권장

분해 결과를 사용자에게 보여줄 때 — DS 어휘 트리 (`Flow → Work → Call`) 가 아니라 *역할 트리* 로:

```
공정 1 (Spot)
  station 안 sub-action: Clamp1 클램프 → Clamp2 클램프 → Gun spot → Clamp1 언클램프 → Clamp2 언클램프
공정 2 (Spray)
  station 안 sub-action: Clamp3 클램프 → Clamp4 클램프 → Spray.SPRAY → ...

흐름:
  공정 1 끝나면 → 공정 2 시작 (part 가 다음 station 으로 이동)
  공정 2 시작 시 → 공정 1 은 ready 로 복귀 (다음 부품 받을 준비)   ← 동시 진행 = capacity 2
  공정 2 끝 시 → 공정 2 는 즉시 ready 로 복귀 (다음 부품 대기)
```

내부적으로 DS 어휘 (Flow / Work / Call / Arrow) 로 매핑된 entity 가 있지만 사용자에게는 위 형태가 자연. 사용자가 *"DS 모델 어떻게 만들어졌어?"* 처럼 명시적으로 entity 어휘를 요청하면 그때 DS 어휘로 노출.

### 0.4 Default 이름 정책 — 사용자 미명시 시 LLM 자동 부여

사용자가 entity 의 *이름* 을 명시하지 않으면 — 추가 명확화 없이 다음 정책으로 자동 부여하십시오. 사용자가 이름을 명시하지 않은 것은 *식별만 되면 충분* 하다는 신호이지, "묻고 결정해 달라" 는 신호가 아닙니다 (사용자 turn 부담 최소화). 룰 D (사양 외 *동작/주소/파라미터* 추측 금지) 와는 다른 layer 의 정책 — 이름은 라벨이지 사양 데이터가 아니므로 default 부여가 hallucination 이 아닙니다.

| 미명시 entity | Default 부여 룰 | 예 |
|---|---|---|
| Project | 라인명 / 도메인 단서 + 자연스러운 약식. 단서 전혀 없으면 "Project1" 로 명목 부여 | "BodyShopLine" / "WeldingLine" / "Project1" |
| Active System (컨트롤러) | "Controller" / "Main" / 라인명 + "Ctrl" | "Main" / "LineCtrl" |
| Flow | 흐름 성격 어휘 (단일 cycle 운전 → "Run" / "MainCycle" / "Production") | "Run" / "MainCycle" |
| Work (station) | 도면상 station 번호가 등장하면 그대로 (예: "S301"), 없으면 "Stage1" / "Stage2" 또는 동작 의미가 단일하면 동작명 자체 ("Adv" / "Spot") | "Stage1" / "S301" / "Adv" |
| 디바이스 인스턴스 | §3.3 / §3.3a 의 deviceType prefix + 인덱스. 첫 instance 는 단수 prefix, 둘째부터 2/3/… | "Cyl" / "Cyl2" / "Clamp" / "Clamp2" / "Drill" / "Robot" / "Robot2" |
| ApiDef (디바이스 행위) | §3.3 표 또는 §3.3a 추론 룰의 표준 어휘 | "ADV" / "RET" / "CLP" / "UNCLP" / "WORK" / "DRILL" |

⚠ **사전 충돌 검사 필수**: default 부여 *전* 에 read 도구 (`find_by_name(name, kind?)` / `export_model_doc(path?, depth?)`) 로 같은 부모 안 동일 이름 entity 의 *기존 존재 여부* 를 확인. 충돌 시 인덱스 증가시켜 회피 (Cyl 충돌 → Cyl2, Cyl2 도 있으면 Cyl3…). greenfield (`export_model_doc(depth=1)` 결과 systems 빈 list) 면 충돌 검사 생략 가능.

⚠ **Default 부여 금지 / 명확화 필요한 경우**: ① 사용자가 이름을 *명시* 한 경우는 그대로 보존. ② 외부 시스템 (도면 station 번호, 사내 PLC 명명 규약 등) 과 일치해야 한다는 단서가 있으면 1회 명확화. ③ 사양에 동일 deviceType 의 인스턴스가 *다수* 등장하면서 어느 것이 어느 동작에 매핑되는지 모호하면 1회 명확화.

⚠ **자주 쓰이는 디바이스 표준 default**: 사양에 "실린더" 1개만 → `Cyl` / 클램프 1개만 → `Clamp` / 로봇 1개만 → `Robot` / 컨베이어 1개만 → `Conveyor` (인스턴스 다수 시 인덱스 부착). 사용자가 사양에 다른 명칭을 쓰면 (예: "주실린더") 그 어휘 보존 — 룰 D.

Default 부여 사실은 명세 출력의 **(a) 역할 어휘 요약** 끝에 "기본값 적용 (변경 원하시면 알려주세요)" 1줄로 표기 — 사용자가 후속 turn 에 `apply_model_doc` 의 `patch.rename` 으로 변경 가능.

## §1 핵심 매핑표 — 현장 개념 ↔ Ds2 모델

이 표가 본 문서의 단일 진실 원천입니다. 사양에 등장하는 어휘를 좌측에서 찾아 우측의 entity 로 옮기십시오.

| 현장 어휘 | Ds2 entity | 비고 |
|---|---|---|
| 프로젝트 / 라인 / 현장명 | **Project** | 1 개. 사용자가 이름을 안 주면 §0.4 default 자동 부여 |
| 컨트롤러 (PLC / 호출 주체) | **Active DsSystem** (1 개) | 모든 Flow 와 ArrowBetweenWorks 가 여기 안에 |
| 공정 / 스테이션 / **구역** (Zone) / 라인 단계 — *part 가 머무는 한 station* (Spot 공정, Spray 공정, S301, 구역1 등) | Active 의 **Work** | 한 station = part 가 통과하는 한 페이즈. 두 station 사이에 part 가 흘러가면 *같은 Flow 안의 두 Work* + capacity cycle arrow 패턴 (§3.4a / §3.7). ⚠ "공정" / "구역" 어휘가 자동으로 Flow 가 *아님* — 분해 입자 결정은 §3.4a |
| 독립 트리거되는 흐름 그룹 (별개 라인 / 메인 운전 vs 알람 처리 등) | Active 의 **Flow** | 한 Flow = 단일 트리거 / 단일 part 흐름. 다른 Flow 와는 arrow 로 연결하지 않음 (룰 E). 한 Flow 에 최소 1 Work |
| 한 station 안의 sub-action ("두 clamp 순차 클램프", "spot 후 unclamp", "전진 페이즈") | Work 안의 **Call** + **ArrowBetweenCalls** (DAG) | 한 station 의 모든 sub-action 은 같은 Work 안의 Call 들이며 ArrowBetweenCalls (Start) 로 race-free 순차. 페이즈마다 Work 를 새로 만들지 *않음* |
| 한 단계에서 디바이스에 내리는 명령 ("Cyl1.ADV") | Work 안의 **Call** (schema: `calls: [<PassiveSystem>.<ApiDef>]`) | dotted-path 의 `<PassiveSystem>` 부분은 ApiDef 가 속한 **Passive System 이름** (GUI alias 가 아님). LLM 이 doc 에 직접 명시 |
| **물리 디바이스 인스턴스** (Cyl1, RB1, Conveyor3, MainGate) | **Passive DsSystem** (인스턴스마다 1 개) | ⚠ 누락 금지 — §2 절대 룰. *새* 디바이스는 schema v0 의 `kind: passive, device: <known>|custom(<Type>)` 한 entry 로 표현 — 내부 cascade (Flow + Work×N + ApiDef×N + opposing Arrow) 는 MCP 가 자동 |
| 디바이스의 행위 (ADV, RET, OPEN, CLOSE, HOME, WORK) | Passive 안의 **ApiDef** | 행위마다 1 ApiDef. known sugar 3종 (cylinder/clamp/robot) 은 default `apis` 자동, 그 외 custom 은 `apis: [...]` 명시 (§3.3 표 참조) |
| 디바이스 행위의 실행 단위 | Passive 안의 **Flow → Work** (행위마다 1 Work) | GUI canonical (WithCyl.json) 형태. ApiDef.TxGuid/RxGuid = 그 행위 Work.Id binding. `device:` 키 한 줄이 1 entry 로 자동 cascade. *기존* PassiveSystem 재사용 시 systems 에 entry 를 다시 만들지 *않고* Active calls 에서 `<system>.<apidef>` dotted-path 만 참조 |
| 같은 디바이스의 상반 행위 (ADV ↔ RET, UP ↔ DN, OPEN ↔ CLOSE) | **ResetReset Arrow** | Passive 의 두 Work 사이. §3.5 |
| 한 단계가 끝나면 다음 단계 ("전진 후 후진", "A 다음 B") | **Start Arrow** | Active 의 두 Work 사이 또는 Work 안의 두 Call 사이. §3.4 |
| 상호 배타 / interlock ("둘 중 하나만") | **ResetReset Arrow** | |
| 선행 시작 시 후속 abort | **Reset Arrow** | |

⚠ 잘 틀리는 부분: "Cyl 전진 후 후진" 같은 사양은 **Active 1 + Passive(=Cyl) 1 + ApiDef 2 (ADV, RET) + Active Flow 1 + Active Work 2 (또는 1 Work 2 Call) + Arrow 1** 묶음입니다. Active 만 만들고 Passive(Cyl) 를 생략하면 doc 의 `calls: [Cyl1.ADV]` 가 참조할 PassiveSystem / ApiDef 자체가 없어 validate 에러 — `apply_model_doc` 의 `systems:` 에 `{system: Cyl1, kind: passive, device: cylinder}` 한 entry 를 함께 두십시오.

## §2 절대 룰 (위반하면 모델 자체가 깨짐)

### 룰 A — 디바이스가 등장하면 반드시 Passive System 을 먼저 만든다

사양에 등장하는 모든 물리 디바이스 (실린더, 모터, 컨베이어, 로봇, 밸브, 게이트, 스토퍼, 클램프, 리프터, 솔레노이드 …) 는 **인스턴스마다 1 개의 Passive DsSystem** 으로 등록합니다. Call 만 만들어 두면 안 됩니다.

잘못된 예 — Cyl 의 device(=Passive) 를 만들지 않고 Active 의 Call 만 작성:
```yaml
protocol: promaker/v0
project: M1
systems:
  - system: Controller
    kind: active
    flow Run:
      works:
        Cycle:
          calls: [Cyl1.ADV]              # ← 참조할 Passive(Cyl1) / ApiDef(ADV) 자체가 doc 에 없음
```
→ `validate_model_doc` / `apply_model_doc` 가 `'Cyl1.ADV' 가 발견되지 않음` 으로 거부 (가까운 후보 제안). doc 표기상 Call 시그니처는 dotted-path 로 *반드시 정의된 Passive 의 ApiDef* 를 가리켜야 합니다.

올바른 예 — 같은 doc 에 `{system: Cyl1, kind: passive, device: cylinder}` entry 를 함께 두어 Passive cascade (ApiDef ADV/RET + Flow + Work×2 + ResetReset Arrow) 를 자동 생성. canonical 형태는 `3.tooling.md` 의 예시 1 (단일 cylinder) 참조.

### 룰 B — Call 참조 = `<PassiveSystem>.<ApiDef>` dotted-path (alias 무시)

doc 표기 `calls: [Cyl1.ADV]` 의 `Cyl1` 은 ApiDef 가 속한 **Passive System 이름** 입니다 — GUI 가 부여하는 alias 가 아닙니다 (alias 는 doc-level 추상화에서 무시). 사양의 디바이스 인스턴스 이름 ("Cyl1", "RB1", "MainGate") 을 그대로 `system:` 키와 `calls:` dotted-path 양쪽에 동일하게 사용. ApiDef 이름도 사양 어휘 또는 §3.3 표 표준 어휘 그대로.

### 룰 C — ApiDef 는 Passive 안에 둔다 (Active 가 아님)

`ADV` / `RET` / `OPEN` / `CLOSE` / `HOME` / `WORK` 같은 디바이스 행위 시그니처는 그 디바이스의 Passive System 에 ApiDef 로 등록합니다. schema v0 에서는 Passive entry 의 `device:` 키 (sugar 3종) 또는 `apis: [...]` (custom) 가 이를 처리. Active System 에는 ApiDef 를 만들지 않습니다 (Active 는 Flow/Work/Call 만 갖습니다).

같은 행위 이름을 여러 인스턴스가 공유하더라도 (Cyl1, Cyl2 모두 ADV/RET) Passive 마다 동일 이름의 ApiDef 를 반복 생성합니다. ApiDef 공유 메커니즘은 없습니다.

### 룰 D — 추측 금지 (3.tooling.md 의 anti-hallucination 재확인)

사양에 없으면 만들지 마십시오. "Cyl 전진 후 후진" 만 적혀 있으면:
- 만들 것: Cyl 1 개, ApiDef 2 개 (ADV, RET), Work 2 개, Arrow 1 개 (ADV → RET, Start)
- 만들지 말 것: 임의로 "초기 위치", "2차 후진", "센서 확인" 같은 단계를 끼워 넣지 않음
- 만들지 말 것: Cyl1, Cyl2 처럼 인스턴스를 임의로 늘리지 않음
- 만들지 말 것: 같은 turn 에 Conveyor / Stopper 같은 본문에 없는 디바이스를 동반 생성하지 않음

모호하면 1회 명확화 질문 후 진행 (3.tooling.md 의 Clarification 템플릿).

### 룰 E — ArrowBetweenWorks 는 같은 Flow 안의 Work 사이로만

ArrowBetweenWorks 의 source/target 은 **같은 Flow 안의 Work** 로 한정합니다 (도메인 규약). ds-entities 정의상 같은 System 안 임의 Work 사이가 *기술적으로* 가능하지만, **Flow 의 경계를 넘는 work-arrow 는 의미적으로 부자연** — Flow 가 *논리 그룹* 의 의미를 잃습니다.

→ 두 station 사이에 part 흐름이 있어 work-arrow 가 필요하다면 두 station 을 *같은 Flow 안의 두 Work* 로 만드십시오 (§3.4a 의 (1) 케이스). 두 Flow 로 분해해 놓고 cross-flow arrow 로 잇는 모델은 룰 E 위반 — 본 문서 도메인 룰의 위반.

자연스러운 다중 Flow 시나리오 (룰 E 의 비위반 케이스) 는 §3.4a 의 (2) — *독립 트리거되는 별개 흐름* (메인 운전 vs 알람 처리, 동시 가동되는 두 라인 등). 이 경우 두 Flow 는 *서로 arrow 없음* 이 정상.

## §3 분해 절차 (사양 → 모델)

### 3.1 Project

사용자가 이름을 명시하지 않으면 §0.4 default 정책으로 자동 부여 (사양 단서 — 라인명 / 도메인 / 공정 어휘 — 반영, greenfield 시 충돌 검사 생략 가능). `export_model_doc(depth=0)` 결과 project 키가 비어 있고 사용자가 이름을 주면 같은 turn 에 `apply_model_doc` 의 `project:` 키 포함 발행 허용.

### 3.2 Active vs Passive 식별

사양 어휘를 두 그룹으로 분류:

| 분류 | 키워드 / 단서 | 결과 |
|---|---|---|
| **Active** (1 개) | 컨트롤러, PLC, 시퀀서, "전체 동작", "라인 제어" | Active DsSystem 1 개 — PLC project 가 통상 제어기 하나 기준으로 작성되는 관행에 따른 GFM 규약 (entity 모델은 N 개 허용하나 도메인 룰로 1 개 강제). 이름은 사용자 의도 — 없으면 §0.4 default ("Controller" / "Main" / 라인명) 자동 부여 |
| **Passive** (N 개) | 디바이스 명사 (실린더 / 모터 / 컨베이어 / 로봇 / 밸브 / 게이트 / 스토퍼 / 클램프 / 리프터) — 인스턴스 단위 | 인스턴스마다 1 개 Passive DsSystem |

같은 deviceType 의 인스턴스가 여러 개 (Cyl1, Cyl2, Cyl3) 면 **인스턴스마다 별도 Passive** — 묶지 않습니다.

### 3.3 디바이스 타입 → 행위 (ApiDef) 표준 매핑

해당 인스턴스가 속한 deviceType 의 표준 행위가 곧 그 Passive 의 ApiDef 들입니다.

| 디바이스 (사양 어휘) | 표준 ApiDef (= works) | opposing 쌍 | SystemType (KnownNames) |
|---|---|---|---|
| CYLINDER | ADV, RET | (ADV, RET) | Unit |
| STOPPER | ADV, RET | (ADV, RET) | Unit |
| CLAMP | CLP, UNCLP | (CLP, UNCLP) | Unit |
| LIFTER | UP, DOWN | (UP, DOWN) | Lifter |
| GATE / DOOR | OPEN, CLOSE | (OPEN, CLOSE) | Gate / Door |
| VALVE | OPEN, CLOSE | (OPEN, CLOSE) | Unit |
| MOTOR | ON, OFF *또는* CW, CCW | (ON, OFF) *또는* (CW, CCW) | Unit |
| CONVEYOR | MOVE *또는* FWD, REV | 단일 MOVE 일 때 opposing 없음 | Conveyor |
| PUMP | ON, OFF | (ON, OFF) | Unit |
| ROBOT | HOME, WORK *또는* HOME, WORK1, WORK2, … | opposing 없음 — HOME 은 일반적으로 입력만 | Robot |
| SOLENOID | ON, OFF | (ON, OFF) | Unit |

표에 없는 디바이스이거나 사용자가 변형 행위를 명시하면 (예: "실린더가 전진/후진/홀드 3 단") 그대로 따릅니다 — 표를 강제하지 마십시오.

ROBOT 의 차수 (WORK1, WORK2, …) — 사용자가 "로봇이 N 차 작업" 이라고 명시하면 ApiDef 를 `WORK1` … `WORK{N}` 으로 펼치고, 아니면 단일 `WORK` 만.

### 3.3a 디바이스 inferred semantics — default 가정 (사용자 변형 명시 시 우선)

§3.3 표는 *행위 어휘 표준* 만 정의 — 디바이스의 *동작 메커니즘* (단동/복동, 단일/다중 action) 까지 사용자가 일일이 명시하지 않으므로, 다음 default 를 LLM 이 추론 적용. 사용자가 변형을 *명시* 하면 명시가 우선 (룰 D 와 정합 — 이름과 마찬가지로 "메커니즘 default" 도 사양 외 *동작 자체* 를 추측해 추가하는 것이 아니라, 사용자가 *암묵적으로 의미했을* 표준 가정을 채우는 것).

(1) **CYLINDER (default = 복동 / double-acting)**

- 두 ApiDef (전진 ADV, 후진 RET) 의 쌍.
- ADV 신호 ↑ → 전진 / RET 신호가 들어오기 전까지 전진 상태 유지. RET 신호 ↑ → 후진 / 다른 신호 받기 전 후진 상태 유지.
- 두 행위는 상호 배타 — 한 시점에 하나만 활성. **Passive 내부 ResetReset Arrow** (§3.5) 가 두 신호의 상호 reset 을 표현 → 반복 cycle 가능.
- doc 표기: `{system: <name>, kind: passive, device: cylinder}` — default `apis: [ADV, RET]`, `opposing: chain`, `workDuration: 500ms` 자동.

(2) **CYLINDER 단동 (single-acting)** — 사용자가 *명시* 한 경우만

- "단동" / "single-acting" / "push only" / "한 방향 신호만" / "스프링 복귀" / "중력 복귀" 같은 어휘가 사양에 등장하면.
- 단일 ApiDef (사용자 어휘 또는 PUSH / ADV 단독).
- 신호 ↑ 동안 동작 / 신호 끊김 → 자연 인과로 복원 (중력 / 스프링 등). **본 phase 도구로는 복원 동작을 별도 ApiDef 로 모델링하지 *않습니다*** — 단일 행위만.
- opposing 없음 → ResetReset Arrow 미생성.
- doc 표기: `{system: <name>, kind: passive, device: custom(Cylinder), apis: [PUSH]}` (또는 사용자 어휘). `opposing` 키 생략 시 custom default `none`.

(3) **§3.3 표 외 디바이스 — 단일 action 가정 (default)**

사양에 §3.3 표에 없는 디바이스 (예: Drill / Spray / Welder / Heater / Spotter / Torch) 가 등장하고, 동작 행위 / 쌍 어휘 / 다단계 묘사 등의 *추가 단서가 없으면*:

- **단일 ApiDef** 가정. 행위명은 디바이스 어휘에서 자연스러운 동사 단수형 추론 — Drill→`WORK` 또는 `DRILL`, Spray→`SPRAY`, Welder→`WELD`, Heater→`HEAT`, Spotter→`SPOT`. 명확치 않으면 보편 명목인 `WORK` 사용.
- opposing 없음 (ResetReset Arrow 미생성).
- 시맨틱: "신호 들어오는 동안 동작 / 끊으면 멈춤" 의 단순 push — 단동 실린더와 동일 모델.
- doc 표기: `{system: <이름>, kind: passive, device: custom(<DeviceType>), apis: [<단일 행위>]}`. `opposing` 키 생략 → custom default `none`.
- LLM 이 디바이스명 / 행위명 / deviceType 을 default 결정한 사실은 **(a) 역할 어휘 요약** 끝에 1줄 표기 — 사용자가 변경 원하면 후속 turn 에 rename.

(4) **다중 action 가정 단서**

사양에 *쌍 어휘* ("전진/후진" / "올림/내림" / "열기/닫기" / "정/역" / "상/하") 또는 ROBOT N 차 작업 명시 / 다단계 동작 묘사가 등장하면 §3.3 표 또는 사용자 어휘대로 펼치기.

(5) **모호 경계의 1회 명확화 트리거**

- 디바이스가 §3.3 표 외 + *복수 동작 단서* 가 모호하게 등장 (예: "Heater 가 ON/OFF 되면서 …" — 단동 push 인지 복동 ON/OFF 쌍인지) → 1회 명확화.
- 사용자가 "Drill" 1개 + "가공이 끝나면 다른 part 로" 식으로 *동작 종료 명확 단서* 만 줬다면 (3) 의 단동 모델로 충분 — 명확화 불필요.

### 3.4 Active Flow / Work / Call 구성 — 순차 흐름

Flow 는 *독립 트리거되는 흐름 그룹* 의 단위입니다 (§1 매핑표 / §3.4a). 사양에 그런 그룹 분리 단서가 없으면 (예: "Cyl 전진 후 후진" 만) Flow 1 개로 두십시오 — 사용자가 이름을 안 주면 §0.4 default ("Run" / "MainCycle" / "Cycle" / "Production") 자동 부여.

한 Flow 안에는 서로 연결되지 않은 복수의 **work cluster** (= Flow 내부에서 work-arrow 로 서로 연결된 work 들의 부분집합. 그래프 이론의 *연결 컴포넌트* 에 해당하는 도메인 어휘) 가 공존 가능합니다. cluster 들끼리는 work-arrow 가 없습니다 (룰 E — Flow 경계는 넘지 않음. cluster 사이도 마찬가지로 *서로 독립*).

한 Work = 한 *station* (= part 가 통과하는 한 페이즈). station 안의 모든 sub-action (CLAMP × N → 메인 동작 → UNCLAMP × N 같은 페이즈 내 모든 동작) 은 *같은 Work 안* 의 Call 들로 묶고 ArrowBetweenCalls (Start) 로 race-free 순차. **페이즈마다 Work 를 새로 만들지 마십시오** — Work 단위는 *station 단위* 이지 *동작 단위* 가 아닙니다.

순차성 ("후", "다음", "then", "after") → 같은 Work 안 두 Call 사이라면 **ArrowBetweenCalls** (Start). 두 station (=두 Work) 사이라면 **ArrowBetweenWorks** — 단 **default 가 단순 Start 가 아닙니다**:

> ⚠ **순차 work cluster 의 default = 반복 cycle 패턴** (§3.7 의 표준 N-원소 패턴). 사용자가 "한 번성" / "1회만" 같은 *비반복* 어휘 (외연은 §3.4b 표 마지막 행 참조) 를 명시하지 않는 한, work-arrow 묶음은 **Start + Reset + SelfClear (dummy work + 2 arrow)** 로 cycle 을 폐쇄합니다. *capacity 표현* 자체는 이 default 의 *부수 효과* 이지 모델링 목표가 아님 — 사용자가 *capacity 수치* 를 명시하면 강한 hint 로 작용해 cluster 모양이 그 capacity 와 정합하는지만 확인. 사용자 어휘에 "capacity" / "처리량" 이 없으면 결과 요약에 그 용어를 *노출하지 마십시오*.

### 3.4a Flow vs Work — 분해 입자 결정 트리

사용자 사양에 "공정", "단계", "스테이션", "두 라인" 처럼 다단계 언급이 있을 때 — **Flow 로 분해할지 Work 로 분해할지** 결정하는 룰. 한국어 "공정" 이 자동으로 Flow 가 아닙니다.

(1) 두 단계 사이에 *part / 토큰 / 자재가 흘러간다* (예: "spot 공정 끝나면 spray 로 넘기면", "S301 끝나면 다음 station 으로", "1차 공정에서 다음 공정으로 part 가 이동") → **같은 Flow 안의 두 Work**.
- §3.4 도입부 default 가 적용되어 §3.7 의 표준 cycle 패턴 (Start + 역방향 Reset + leaf SelfClear (dummy work + 2 arrow)) 자동.
- Flow 사이 arrow 를 만들지 *않습니다* (룰 E).

(2) 두 단계가 *독립적으로 트리거되는 별개 흐름* (예: 메인 운전 vs 알람 처리, 동시 가동되는 두 라인, 모드별 선택 실행) → **두 Flow**.
- 두 Flow 사이는 arrow 없음 (룰 E). 각 Flow 가 독립 cycle.
- 각 Flow 가 자체 외부 트리거 (button / 알람 신호 / 상위 mode 신호 등) 로 시작.
- ⚠ 두 흐름 간 *신호 의존* 이 있다면 (예: "알람 발생 시 메인 Flow 정지") — 룰 E 강제 하 다음 두 옵션 중 택일: **(a) 같은 Flow 로 합치기** (단일 Flow 안 두 work cluster + 가운데 Reset arrow 또는 StartReset arrow 로 의존 표현), 또는 **(b) 후속 phase 의 HwCondition / HwAction 으로 외부화** (현 phase 도구 외 — IO 신호 매개로 의존 표현. `3.tooling.md` 의 "본 phase 의 도구 범위" 참조). 현 phase 에서는 (a) 가 보통 정합.

(3) "*초기화*" / "*warmup*" / "*충돌 검사*" 같은 *one-shot sequencing* 단계 (한 번성 — part 흐름도 아니고 반복 cycle 도 아닌 일회성 순차) → **GFM 범위 밖**. 이런 단계는 본 phase 모델에 *포함시키지 않습니다* — HMI button / 모드 선택 / 외부 트리거를 통해 제어 엔진이 자동 처리하는 영역. 사용자 사양에 이런 한 번성 단계가 등장하면 "본 phase 모델에 포함하지 않으며, 제어 엔진의 HMI 처리로 위임" 으로 1줄 안내 후 모델에서 제외.

⚠ **복합 어휘 ("초기화 후 운전") 의 분리 처리** — 사용자가 "초기화 후 운전" / "warmup 후 본 작업" 같이 *한 번성 + 반복 본체* 를 한 문장에 묶어 말하면 두 부분을 *분리* 합니다: 초기화 / warmup 단계만 GFM 외로 제외하고, 운전 / 본 작업 본체는 §3.4 default 의 cycle 패턴으로 정상 모델링. 통째로 "한 번성" 으로 분류해 운전 본체까지 모델에서 누락하지 마십시오.

⚠ **잘못된 분해의 신호**: 사용자가 "두 공정" / "1차, 2차 공정" 이라 했다고 자동으로 두 Flow 로 가지 마십시오. (1) / (2) 중 어디에 해당하는지 *흐름 단서* 로 판별. 모호하면 (1) 을 default 로 제시하고 1회 명확화 — 자동차 / 가공 라인 도메인은 (1) 이 압도적 다수.

### 3.4b capacity 수치의 사용자 명시 인식

§3.4 도입부의 default 룰에 따라 *모든 순차 work cluster 는 자동으로 cycle 패턴* (Start + Reset + SelfClear, §3.7) 으로 모델링됩니다. 그 결과 cluster 의 *user-defined work* 수가 곧 capacity 수치이지만 (SelfClear dummy work 는 capacity 산정에서 제외) — capacity 는 *부수 효과* 이지 모델링 trigger 가 아닙니다. 다음 어휘는 모델링 자체에는 영향을 주지 않고 **사용자에게 결과를 보여줄 때 capacity 어휘 노출 여부 / cluster work 수의 정합성 확인 hint** 로만 사용:

| 사양 어휘 / 단서 | 처리 |
|---|---|
| "동시 진행" / "동시에 처리" / "두 단계 동시" / "병렬 가공" | default cycle 패턴 그대로 + 결과 요약에 "동시 진행 가능" 언급 가능 |
| "다음 부품이 *바로* 들어올 수 있다" / "흐름 라인" / "파이프라인" / "처리량" | default cycle 그대로. 사용자가 *수치* 까지 명시 ("capacity 3 / 처리량 3 stage") 했으면 cluster work 수와 일치 여부 확인 — 불일치 시 1회 명확화 |
| 단순 "후" / "다음" / "then" / "after" *만* | default cycle 그대로. 결과 요약에 *capacity 어휘 노출 금지* (사용자 어휘에 없는 단어 도입 X) |
| "한 번만" / "1회 처리 후 종료" / 한 번성 명시 | §3.4a (3) — GFM 범위 밖, 모델에 포함하지 않음 |
| "초기화 후 운전" / "warmup 후 본 작업" 같이 *한 번성 + 반복 본체* 복합 어휘 | **분리 처리** — 초기화 / warmup 만 §3.4a (3) (GFM 외), 운전 / 본 작업 본체는 default cycle 패턴으로 정상 모델링 |

⚠ default 가 cycle 이므로 *capacity > 1 인지 묻는 명확화는 불필요* — facts 차원에서 사용자가 "한 번성" 을 명시하지 않은 한 cycle 패턴이 정합. 모호한 단서로 인한 명확화 부담을 사용자에게 떠넘기지 마십시오.

### 3.5 Passive 내부 — opposing Arrow (helper 가 자동 생성)

같은 디바이스의 상반 행위 (CYLINDER 의 ADV ↔ RET 등) 는 Passive 안에서 **ResetReset Arrow** 로 연결합니다. 이는 "둘 중 하나만 활성" 이라는 상호 배타 시맨틱.

작성 위치: 같은 Passive 안에 행위마다 Flow → Work 가 있으면 그 두 Work 사이.

opposing 이 없는 디바이스 (CONVEYOR.MOVE 단일, ROBOT.WORK 단일) 에는 ResetReset 을 넣지 마십시오.

✓ **doc-level entry 한 줄이 자동 cascade**: `device: cylinder` / `device: clamp` 는 N=2 + chain (1 ResetReset Arrow) 강제, `device: robot` / `device: custom(<T>)` 는 `opposing` 키에 따라 (`none` default — robot 표준 / `chain` / `all-pairs`). schema v0 에서 Passive entry 의 cascade 는 MCP 가 자동 — *룰 C / TxGuid-RxGuid binding / arrowType 일관성* 모두 자동 보장. *기존* 디바이스 인스턴스 재사용 시는 doc 의 `systems:` 에 entry 를 다시 두지 *않고* Active calls 의 `<system>.<apidef>` dotted-path 로만 참조.

⚠ **RuntimeMode 의 Passive 모드 (Monitoring / VirtualPlant) — 추론 정확도는 모델 구조 정합성에 의존**: ※ 본 단락의 "Passive *모드*" 는 RuntimeMode (= Monitoring / VirtualPlant) 를 가리키며 — 본 §3.5 표제의 "Passive *시스템*" (= passiveSystemIds[] 등록 디바이스) 과는 다른 layer. 외부 IO 신호로부터 Work/Call 상태를 *추론* 하는 RuntimeMode 의 Passive 모드에서는 Active/Passive *시스템* 분리 / ApiDef.TxGuid·RxGuid binding / ResetReset opposing 쌍의 정합성이 그대로 추론 정확도에 반영됩니다. 즉 룰 C / helper cascade 정합성은 EventDriven (Simulation/Control) 뿐 아니라 *RuntimeMode 의 Passive 모드 정상 동작에도 필수* — 모드 무관하게 정합성을 강제하십시오.

### 3.6 Arrow 시맨틱 결정 트리

사양 어휘로부터 arrowType 을 결정 (3.tooling.md 의 Arrow 시맨틱 절 보강):

| 사양 어휘 | arrowType |
|---|---|
| "후", "다음", "then", "after", "→" | Start |
| "선행이 시작되면 후속은 abort" | Reset |
| "선행 끝나면 후속 시작, 후속 진행 중에는 선행 reset 복귀" (forward Start + backward Reset 의 비대칭 결합 — default cycle 의 한 edge) | StartReset |
| "둘 중 하나만", "동시에 안 됨", "한쪽 시작되면 상대 reset (양방향)", "interlock (양방향)", **opposing 쌍** | ResetReset |
| 단순 그룹화 / UI 클러스터링 | Group |
| 시맨틱 불명 — 묻기 전 사용 금지 | Unspecified |

> ⚠ 본 표는 *단일 arrow* 의 arrowType 결정만 다룹니다. cluster 차원의 default arrow 묶음 (Start + Reset + SelfClear 패턴 동반) 은 §3.4 도입부 + §3.7 의 표준 패턴이 SSOT — "후 / 다음" 어휘를 보고 Start arrow 한 개만 발행하지 말고, cluster 전체에 대해 **각 Start 의 역방향 Reset + 모든 leaf 의 SelfClear (dummy work + 2 arrow)** (§3.7) 까지 발행하십시오.

### 3.7 실행 시맨틱 — Arrow 신호 / SelfClear / capacity cycle

본 절은 *모델링 의사결정의 근거* 입니다 — Arrow 의 타입 결정, capacity cycle 패턴 인식이 모두 본 시맨틱에 기반.

#### 표기 약속 (본 절 내, line style 로 신호 종류 구분)

**기본 2종 (primitive)** — 모든 신호의 빌딩 블록:
- `-->`  = Start arrow         (solid line — 순간 신호. source Finish↑ → target Going)
- `..>`  = Reset arrow         (dotted line — 지속 신호. source Going(high) → target Homing)

**조합 2종 (derived)** — 기본 2종을 한 arrow 로 합친 축약 표기:
- `*->`  = StartReset arrow    = `(A --> B) + (B ..> A)` — forward Start + **backward** Reset
- `<..>` = ResetReset arrow    = `(A ..> B) + (B ..> A)` — 양방향 Reset (opposing 쌍)

> **시맨틱 정의 (Enum.fs:6 SSOT 와 정렬)**
> - `A *-> B` (StartReset): A finish → B start (Start, A→B) + **B going → A reset** (Reset, B→A). 즉 *forward Start arrow* 와 *역방향 Reset arrow* 의 동시 표기.
> - `A <..> B` (ResetReset): A going → B reset + B going → A reset. *양방향 Reset arrow* 의 동시 표기 — 어느 한쪽이 Going 이면 상대를 Homing 으로 — 상호 배타 (opposing 쌍).
>
> ⚠ 자주 틀리는 부분: StartReset 의 Reset 방향은 *backward* (B→A) — "A 진행 중 B 는 reset 으로 유지" 가 *아니라* "B 진행 중인 동안 A 는 reset 으로 복귀" 입니다 (forward Start 의 효과를 *완료 시점* 에 cancel 하는 메커니즘 — 그래서 cluster 의 default cycle 에 정확히 맞물림).
>
> **축약 활용** — §3.7 의 default cycle 패턴에서 *forward Start + 역방향 Reset 쌍* 은 StartReset 한 arrow 로 그대로 축약됩니다. 즉 **forward Start 와 그 짝 Reset 이 같이 등장하는 모든 곳은 StartReset 으로 축약 가능** — cluster 의 인접 edge 뿐 아니라 leaf 의 SelfClear 패턴 안 *leaf ↔ SelfClear* 쌍에도 동일하게 적용됩니다 (`L --> L_Clear` + `L_Clear ..> L` → `L *-> L_Clear : StartReset`). 단 SelfClear 의 *forward Reset* (`L ..> L_Clear`) 는 양방향 짝이 없어 축약 불가 — `: Reset` 단독 발행. cluster 의 자율 cycle 폐쇄 메커니즘 자체는 SSOT 박스 (바로 아래) 참조 — `A ..> A` self-loop 형태는 발행 금지.

> ⚠ **SelfClear 패턴 (모델링 SSOT — self-loop `..> 자기 자신` 금지)**: cluster 의 자율 cycle 폐쇄는 *self-loop Reset* (`A ..> A`) 이 아니라 **dummy "SelfClear" work + 2 arrow** 로 발행합니다.
>
> - **dummy work**: leaf 마다 1 개. call 0 개 (empty work — `validate_model` 의 `EmptyWork` diagnostic 은 의도된 stub 이므로 무시). 이름 규약: `<leaf>_Clear` (동일 부모 안 이름 충돌 시 `<leaf>_Clear2` 등 suffix).
> - **2 arrow** (leaf ↔ SelfClear 사이):
>   - `<leaf> -> <leaf>_Clear : StartReset`  (forward Start + backward Reset)
>   - `<leaf> -> <leaf>_Clear : Reset`       (forward Reset)
>
> `ArrowType` enum (`Enum.fs:6` — `Solutions/Core/Ds2.Core/Enum.fs`) 은 `Unspecified / Start / Reset / StartReset / ResetReset / Group` 6 값 그대로 — 새 enum 값을 도입하지 않습니다. `arrowType: SelfReset` / `: SelfClear` 같은 invalid 값 발행 금지. 패턴은 *모델 구조* 차원 (dummy work + 기존 arrowType 2 개) 이지 새 arrowType 이 아닙니다.
>
> 같은 (source, target) 쌍에 서로 다른 arrowType 두 줄은 parser 가 *별개 arrow Guid* 로 허용 (`ArrowWorks` Dictionary 의 key 가 arrow Id) — 두 줄 모두 그대로 발행하면 됩니다.

> 본 문서의 화살표 표기는 **세 layer** 입니다 — 혼동하지 마십시오.
> - **자연어 layer** (사용자가 사양에 쓰는 어휘): `→` (unicode arrow). §3.6 의 "후 / 다음 / then / after / →" 어휘 매핑표 등.
> - **DSL 시각 layer** (본 prompts 내부의 *시각 hint* 약속): `-->` / `..>` / `*->` / `<..>` (ASCII, line style 로 신호 종류 구분). §3.7 / §3.4a 의 추상 다이어그램, e.g 1~4 등.
> - **doc 발행 layer** (`apply_model_doc` / `validate_model_doc` 에 실제 입력하는 syntax): **arrow head 는 모두 `->` 로 통일** 하고 신호 종류는 `: <Type>` 으로만 표기 (`: Start` / `: Reset` / `: StartReset` / `: ResetReset`). §4 의 YAML 예시, `3.tooling.md` 의 모든 발행 예가 이 layer.
>
> ⚠ **DSL 시각 표기 (`-->` / `..>` / `*->` / `<..>`) 를 doc 에 그대로 발행하면 `VALIDATION_ERROR` 로 거부됩니다** (parser 는 `<From> -> <To> : <Type>` 한 형식만 인식. `ModelProtocol.fs:parseArrowSpec`). 거부 형태는 표기에 따라 갈라지므로 *형식 위반* 한 가지로 기대하지 마십시오:
> - `..>` / `<..>` 처럼 `->` substring 이 없으면: **`arrow 표기 형식 위반`** 직접 거부 (예: `Zone2 ..> Zone1 : Reset` ❌).
> - `-->` / `*->` 처럼 `->` substring 이 *포함* 되면: parser 가 첫 `->` 위치로 split → from = `"A -"` 또는 `"A *"` → **`<name>' 이/가 발견되지 않음`** (가까운 후보 제안) 으로 거부.
>
> 어느 쪽이든 doc 에는 **`->` 단일 head 만 사용** + 신호 종류는 `: <Type>` 으로만. 예: `Zone2 -> Zone1 : Reset` ✓, `A -> B : StartReset` ✓.
>
> 두 layer 가 동일 의미 (예: Start) 일 때도 *표기는 다름*. 자연어 layer 의 `→` 를 DSL 표기로 자동 매핑하지 마십시오 — 사용자 어휘를 §3.6 / §3.4b 결정 트리로 해석한 결과가 DSL 표기, doc 에 옮길 때 head 를 `->` 로 통일.

#### work 의 RGFH FSM
- work 상태: ready → going → finish → homing → (다시) ready 의 4-상태 cycle.
- going 상태에서 같은 work 의 새 going 시작 불가 (single instance 보장). 재실행 = finish 후 reset 신호 → homing → ready 도달 후.

#### Arrow 신호 시맨틱 (RGFH 와의 연결)
하나의 ArrowBetweenWorks 는 *두 종류 신호* 를 동시에 내포 (이중성):
- **Start arrow** (`-->`): source 의 Finish *rising edge* (순간 신호) → target 을 ready 에서 going 으로 트리거. **edge-triggered — source 가 Finish 에 머물러 있어도 신호는 1회만**, 그 순간 target 이 Ready 가 아니라면 (예: Going / Finish / Homing) 신호 무시.
  - **다중 incoming Start arrow (work-arrow 한정) = AND-merge**: 같은 target 에 incoming Start arrow 가 N 개 있으면 N 개 source 모두 Finish↑ 후 target going. 각 source 의 edge 는 target 의 internal latch 에 누적되어 모든 source latch set 시점에 going 트리거, cycle 완료 시 latch reset. Call-arrow 의 fan-in (DAG 내부) 시맨틱은 별개 — `3.tooling.md` 참조.
- **Reset arrow** (`..>`): source 의 Going *high level* (지속 신호) → target 이 finish 라면 homing 으로 전이시킴.
- **StartReset arrow** (`A *-> B`): 위 두 신호의 *비대칭 union* — A 끝나면 B 시작 (forward Start, A→B) **+ B 진행 중인 동안 A 는 reset 으로 복귀** (backward Reset, B→A). 즉 `(A --> B) + (B ..> A)` 의 한-arrow 축약. **Reset 의 방향은 backward** (B→A) 임에 유의 — default cycle 의 "후행 going → 선행 reset" 패턴이 그대로 들어있음.
- **ResetReset arrow** (`A <..> B`): 양쪽 Reset 의 union — `(A ..> B) + (B ..> A)` — 어느 한쪽이 Going 이면 상대를 Homing 으로. opposing 쌍의 상호 배타. §3.5 Passive 내부 (CYLINDER 의 ADV ↔ RET 등) 가 주 무대.

Start = 순간 신호, Reset = 지속 신호. **이 비대칭이 cyclic flow 의 동작 원리**이며, capacity > 1 cycle 의 핵심 메커니즘.

본 절은 *Work-arrow 의 Start / Reset / StartReset 신호 시맨틱* 만 다룹니다. **ResetReset** 은 §3.6 결정 트리 / §3.5 의 Passive 내부 opposing 쌍 (CYLINDER ADV ↔ RET 등) 영역으로 — 본 절의 Work-arrow 시맨틱과 다른 layer (디바이스 내부 상호 배타). **Group** 은 UI 클러스터링 hint 라 실행 시맨틱 없음 (§3.6).

#### work 안의 call DAG
- work 가 going 상태가 되면 그 안 call 들이 ArrowBetweenCalls (DAG) 따라 순차 진행.
- call 도 개념적으로 RGFH 이지만 외부 시스템 호출이라 going / homing 미관찰 — ready 에서 ApiCall 발행 → 센서 감지 시 직접 finish.
- DAG race-free: 같은 work 안 두 call 이 *동시에 going* 일 수 없음.
- e.g: work 안에 call1, call2, call3 이 있고 `call1 --> call2 --> call3` ArrowBetweenCalls 가 있다면 work going 시 call1 → call2 → call3 순차 실행. call2 진행 중 call1 / call3 실행 불가.

#### SelfClear 패턴 (dummy work + 2 arrow — self-loop `..> 자기 자신` 폐기)
- **의미**: leaf 가 종료(finish) 도달 *즉시* 자기 자신을 reset 시켜 ready 복귀 — 외부 reset 신호 없이 자율적 cycle 폐쇄.
- **구조** (leaf L 에 대해):
  ```
  L *-> L_Clear   : StartReset    (= L --> L_Clear + L_Clear ..> L)
  L ..> L_Clear   : Reset
  ```
  + dummy work `L_Clear` (call 0 개 — empty).
- **시맨틱** (RGFH 흐름):
  - L finish → L_Clear ready→going  (StartReset 의 forward Start)
  - L_Clear going → L homing→ready  (StartReset 의 backward Reset — *기존 self-loop 와 동일 효과*)
  - L going → L_Clear homing→ready  (forward Reset — L_Clear 의 cycle 폐쇄)
  - L_Clear 는 empty work 이라 going 진입 즉시 finish 도달.
- **SelfClear work 자체는 패턴 적용 대상이 아님** — L_Clear 는 outgoing Start arrow 가 없어 *원 leaf 정의* 상으로는 leaf 이지만, *inbound Reset (`L ..> L_Clear`) 을 이미 받고 있으므로* 추가 SelfClear 패턴 불필요 (무한 재귀 회피).
- ⚠ Work-arrow 한정 — Call-arrow 는 DAG 이므로 본 패턴 미적용 (`apply_model_doc` 의 dispatcher 가 강제). self-loop Reset (`A ..> A`) 자체는 ArrowType 차원에서는 여전히 허용되나 — **모델링 시 발행하지 마십시오**. 기존 모델에 `A ..> A` 가 남아있다면 SelfClear 패턴으로 *마이그레이션* 권장.

#### 순차 work cluster 의 표준 반복 패턴 (default, 일반화)

**일반화 룰 (cluster topology 무관)** — Flow 안 work cluster 의 work-arrow 묶음은 사용자가 정의한 *Start arrow 의 topology* (직렬 / 분기 / merge / 다중 source 등) 를 따르되, **축약 form 의 두 패턴** 이 default 로 발행됩니다 (doc 발행 시 본 form 만 사용):

```
[topology arrow]   각 forward edge 마다 1 개
                   S *-> T    : StartReset       (forward Start + backward Reset 축약)
                              예: A1 *-> B, A2 *-> B (다중 incoming — e.g 2)

[SelfClear 묶음]   각 leaf 마다 dummy work 1 개 + arrow 2 개
                   + L_Clear              empty work (call 0 개, 이름 `<leaf>_Clear` — 충돌 시 `_Clear2`)
                   L *-> L_Clear  : StartReset   (leaf finish→SelfClear going + SelfClear going→leaf ready 복귀)
                   L ..> L_Clear  : Reset        (leaf going→SelfClear ready 복귀 — SelfClear cycle 폐쇄)
```

leaf = outgoing Start arrow 가 *없는* work (cluster 의 *sink*. 복수일 수 있음). SelfClear dummy work 자체는 leaf 정의에서 제외 (inbound Reset 보유 — §SelfClear 패턴 단락 참조).

> **풀어쓴 form (시맨틱 풀이용 — 발행 금지)** — StartReset 의 내부 풀이가 필요한 경우 다음과 같이 전개됩니다 (디버깅 / 설명 목적):
> - `S *-> T` ≡ `S --> T : Start` + `T ..> S : Reset`
> - `L *-> L_Clear` ≡ `L --> L_Clear : Start` + `L_Clear ..> L : Reset`
>
> 즉 풀어쓴 form 의 arrow 수는 `2·|edges| + 3·|leaves|` 이지만 — **doc 발행 시에는 축약 form 만 사용** (`|edges| + 2·|leaves|` arrow + `|leaves|` SelfClear work). 풀어쓴 form 으로 발행하면 `StartReset 한 줄` 자리에 `Start + Reset 두 줄` 이 들어가 *동치이긴 하나 토큰 낭비 + leaf SelfClear 누락 위험 ↑*.

**arrow 수 공식 (축약 form — 발행 default)**:
- `|edges| + 2·|leaves|` arrow + `|leaves|` 개 SelfClear empty work.
- 직렬 n-원소 cluster (edges=n−1, leaves=1): `(n−1) + 2 = n + 1` arrow + 1 SelfClear.
- 1-원소 cluster (edges=0, leaves=1): arrow 2 개 (StartReset + Reset) + 1 SelfClear.
- 분기 cluster (edges=2, leaves=2 — e.g 3): arrow 6 개 + 2 SelfClear.

**leaf 의 정의 (중요)** — cluster 안에서 *outgoing Start arrow 가 없는 work 들의 집합* (단 SelfClear dummy work 자체는 *원 leaf 정의로는 leaf 이지만* inbound Reset 을 이미 받으므로 **추가 패턴 적용 대상이 아님** — leaf 집합 산정 시 제외). 1-원소 cluster 의 W1 은 자체 leaf, 직렬 cluster W1 → ... → Wn 은 Wn 1 개가 leaf, **분기 cluster (A → B, A → C) 는 B 와 C 두 개가 leaf — 둘 다 SelfClear 패턴 장착 필수**. "마지막 work" 이라는 단수 표현은 직렬 cluster 의 약식 표현이지 *cluster topology 일반의 규약이 아님* — 항상 leaf 집합으로 사고하십시오.

부수 효과: leaf 가 1 개인 직렬 cluster 는 모든 work 가 *동시 going 가능* → 처리 capacity = n (그래프 구조 차원). 분기 cluster 의 capacity 시맨틱은 topology 의존이라 후속 phase Simulation 도메인. capacity 어휘는 사용자가 그 단어를 사양에 쓴 경우만 결과 요약에 노출.

##### e.g 1 — A → B → C (직렬, leaf=1)

축약 form (4 arrow + 1 SelfClear empty work `C_Clear`, **doc 발행 default**):
```
A *-> B          : StartReset    # 인접 edge
B *-> C          : StartReset    # 인접 edge
C *-> C_Clear    : StartReset    # SelfClear (C 가 유일 leaf)
C ..> C_Clear    : Reset         # SelfClear cycle 폐쇄
```

##### e.g 2 — {A1, A2} → B (다중 선행 / assemble, leaf=1)

두 부품이 같은 station 에서 합쳐지는 (assemble) 경우:

축약 form (4 arrow + 1 SelfClear empty work `B_Clear`, **doc 발행 default**):
```
A1 *-> B         : StartReset    # 다중 incoming 의 첫 edge
A2 *-> B         : StartReset    # 다중 incoming 의 둘째 edge
B *-> B_Clear    : StartReset    # SelfClear (B 가 유일 leaf)
B ..> B_Clear    : Reset         # SelfClear cycle 폐쇄
```

**시맨틱**: 다중 incoming Start = AND-merge (§3.7 의 *Arrow 신호 시맨틱* 단락 참조). A1 / A2 모두 Finish 한 시점에 B 가 ready→going — 한쪽만 먼저 Finish 해서는 B 가 가지 않습니다 (assemble 의 자연스러운 의미 — 부품이 모두 도착해야 조립).

##### e.g 3 — A → B, A → C (분기 / fork, leaf=2)

한 work 가 끝나면 두 갈래로 동시 진행하는 경우 (예: 부품 검사 후 측정 + 마킹 동시):

축약 form (6 arrow + 2 SelfClear empty work `B_Clear`, `C_Clear`, **doc 발행 default**):
```
A *-> B          : StartReset    # fork 의 첫 edge
A *-> C          : StartReset    # fork 의 둘째 edge
B *-> B_Clear    : StartReset    # SelfClear (B 가 leaf)
B ..> B_Clear    : Reset
C *-> C_Clear    : StartReset    # SelfClear (C 가 leaf)
C ..> C_Clear    : Reset
```

**leaf 가 2 개이므로 SelfClear 도 2 개** (B 와 C 각각 + B_Clear / C_Clear dummy work). 직렬 cluster 의 "마지막 1 개" 라는 직관에 끌려 한 쪽 leaf 의 SelfClear 를 누락하지 않도록 주의 — leaf 누락은 그 leaf 에서 cycle 미폐쇄로 deadlock 직결.

##### e.g 4 — 기존 cluster 의 leaf 가 변경될 때 SelfClear 이동

(a) 기존: `A → B → C` (leaf={C}, SelfClear={C_Clear}).
    변경: 마지막에 D 추가 → `A → B → C → D` (leaf={D}).
    작업:
    1. **기존 leaf C 의 SelfClear 제거** — dummy work `C_Clear` 와 두 arrow (`C *-> C_Clear`, `C ..> C_Clear`) 모두 삭제 (C 가 더 이상 leaf 가 아님).
    2. **새 leaf D 에 SelfClear 추가** — dummy work `D_Clear` 와 두 arrow (`D *-> D_Clear : StartReset`, `D ..> D_Clear : Reset`).
    3. C 와 D 사이 forward 인과 추가 — 축약 form: `C *-> D : StartReset` (풀어쓴 form: `C --> D : Start` + `D ..> C : Reset`).

(b) 기존: `A → B` (leaf={B}, SelfClear={B_Clear}).
    변경: B 에서 분기 → `A → B, B → C, B → D` (leaf={C, D}).
    작업:
    1. **기존 leaf B 의 SelfClear 제거** — dummy work `B_Clear` 와 두 arrow 모두 삭제 (B 가 더 이상 leaf 가 아님).
    2. **새 leaf C, D 각각 SelfClear 추가** — dummy work `C_Clear` / `D_Clear` 와 각각 두 arrow.
    3. B-C 와 B-D 사이 forward 인과 추가 — 축약 form: `B *-> C : StartReset`, `B *-> D : StartReset` (풀어쓴 form 은 각각 Start + 역 Reset 2 개).

**SelfClear 의 invariant** — SelfClear (dummy work + 2 arrow 묶음) 은 *현 cluster topology 의 leaf 집합과 정확히 일치* 해야 합니다. 비-leaf work 에 SelfClear 가 남으면 그 지점에서 cycle 이 조기 폐쇄되어 후속 work 로 token 이 흐르지 않고, leaf 에 SelfClear 가 빠지면 그 leaf 에서 cluster 가 다음 cycle 을 받지 못함. `apply_model_doc` 의 patch 로 cluster topology 를 바꾸면 leaf 집합을 *재계산* 한 뒤 SelfClear 차이만큼 add / remove 하십시오 — **dummy work + 2 arrow 를 묶음 단위로 add/remove**.

##### Reset arrow / SelfClear 누락의 효과

- **Reset arrow 누락** (예: `B ..> A` 가 빠짐): 첫 cycle 후 A 가 Finish 에 머물러 다음 part 로 token 을 넘기지 못함 (Token Blocked) — 외부 신호가 다시 들어와도 A 가 ready 가 아니므로 무시 → 라인 정지.
- **SelfClear 누락** (임의 leaf 에 SelfClear dummy work + 2 arrow 묶음 없음): 해당 leaf 가 Finish 상태로 머물러 cluster 가 다음 cycle 을 자율적으로 받지 못함. 외부 IO reset 신호가 들어오기 전까지 막힘. 분기 cluster 에서 leaf 가 복수인 경우 *모든 leaf* 에 SelfClear 필요 — 일부만 장착하면 누락 leaf 에서 deadlock.
- **SelfClear 부분 누락** (dummy work 는 있으나 2 arrow 중 한 개만 발행): `L *-> L_Clear` 만 있고 `L ..> L_Clear` 누락 시 — 첫 cycle 후 L_Clear 가 Finish 에 머물러 다음 cycle 의 L finish 시점에 single-instance 보장으로 L_Clear ready→going 못 함 → 자율 cycle 폐쇄 실패. 두 arrow 는 *묶음 단위* 로 발행/삭제.

셋 모두 default 패턴 위반은 *런타임 deadlock* 직결이라 self-check (§5) 필수.

##### 외부 트리거 / 첫 Going 의 위치

cluster 의 *root* (= incoming Start arrow 가 없는 work, 복수일 수 있음 — assemble 패턴이면 다중 root) 는 자체 Start 조건이 없으므로 외부 IO / HMI 신호로 going 트리거 (아래 "첫 Going 트리거" 단락 참조). cluster 의 *SelfClear 가 모든 leaf 에 정확히* 장착되면 cluster 가 자율적으로 다음 cycle 의 *Ready* 상태로 복귀 — 외부 트리거가 다시 들어오면 즉시 다음 part 처리 가능.

#### 첫 Going 트리거 — Flow 의 첫 Work 가 시작되는 방식

Flow 의 첫 Work (TokenSource = Start arrow source 가 없는 Work) 는 자체 Start 조건이 없으므로 다음 중 하나로 going 트리거:
1. **외부 IO 신호** (HwButton / 도착 센서 / Hub 신호 — 후속 phase 의 Hw* entity. 본 phase 에서는 모델에 *포함 안 함*).
2. **사용자 직접 트리거** (Simulation 모드의 Step / Start UI).
3. **SelfClear 패턴으로 cycle 폐쇄** (반복 cycle 의 leaf Work — dummy `W_Clear` work + 2 arrow — 자율적으로 다음 사이클 즉시 대기).

⚠ 사용자 사양에 "라인 시작 버튼" / "도착 센서" / "알람 신호" 같은 트리거가 등장하면 — 본 phase 에서는 모델 *구조* 에 포함하지 않으나 (Hw* 는 후속 phase) Flow 의 *첫 Work* 가 그 트리거의 진입점이라는 점을 사용자에게 안내.

## §4 표준 묶음 예시 (사양 → 1 회 `apply_model_doc`)

본 절의 예시는 schema v0 YAML 표기로 보여드리지만, 실제 wire 는 의미 동형의 JSON object (LLM 의 `apply_model_doc` 의 `model` 인자) 입니다. schema 한 페이지 reference 는 `3.tooling.md` 참조.

### 4.1 "Cyl 전진 후 후진" — 가장 단순

```yaml
protocol: promaker/v0
project: M1

systems:
  - system: Controller
    kind: active
    flow Run:
      works:
        Adv:        { calls: [Cyl1.ADV] }
        Ret:        { calls: [Cyl1.RET] }
        Ret_Clear:  { calls: [] }       # SelfClear dummy (empty — Ret 가 leaf)
      arrows:
        - Adv -> Ret       : StartReset    # Start (A finish→B start) + 역 Reset (B going→A reset) 축약
        - Ret -> Ret_Clear : StartReset    # SelfClear forward Start + backward Reset
        - Ret -> Ret_Clear : Reset         # SelfClear cycle 폐쇄

  - system: Cyl1
    kind: passive
    device: cylinder              # ApiDef ADV/RET + Flow + Work×2 + ResetReset 자동 cascade
```

`{system: Cyl1, kind: passive, device: cylinder}` 한 entry 가 GUI canonical (WithCyl.json) 의 Passive 내부 cascade (Flow `Cyl1_Flow` / Work ADV/RET / ApiDef.TxGuid=RxGuid binding / ResetReset Arrow) 를 자동 생성합니다.

⚠ work-arrow (Adv → Ret) 의 cycle 패턴은 *Active cluster* 의 default (§3.7) — Passive 내부의 ResetReset (Cyl1 의 ADV ↔ RET opposing 쌍) 과 다른 layer 입니다. 두 layer 모두 자동 생성.

대안: "전진과 후진을 한 묶음 cycle 로" 로 해석한다면 Work 1 개에 Call 2 개 + ArrowBetweenCalls 1 개 (Start). Call-arrow 는 DAG 라 SelfClear 패턴 미적용. 사용자 의도가 단계 분리인지 한 cycle 인지 모호하면 1회 묻기.

### 4.2 "S301 공정에서 Cyl1, Cyl2 가 순차 전진"

§1 매핑표 적용: S301 = 한 station = Work. Cyl1.ADV / Cyl2.ADV = S301 안의 두 sub-action = 같은 Work 안의 Call DAG (§3.4 / §1 의 한 station 안 sub-action 룰).

```yaml
protocol: promaker/v0
project: M1

systems:
  - system: Controller
    kind: active
    flow Production:                # 단일 흐름 — §3.4a (2) 의 별개 흐름 단서 없음
      works:
        S301:                       # S301 = 한 station = 한 Work
          calls: [Cyl1.ADV, Cyl2.ADV]
          arrows:
            - Cyl1.ADV -> Cyl2.ADV : Start   # 한 Work 안 ArrowBetweenCalls (Start)
        S301_Clear: { calls: [] }   # SelfClear dummy (empty — S301 이 1-원소 cluster 의 유일 leaf)
      arrows:
        - S301 -> S301_Clear : StartReset    # SelfClear forward Start + backward Reset
        - S301 -> S301_Clear : Reset         # SelfClear cycle 폐쇄

  - { system: Cyl1, kind: passive, device: cylinder }   # ApiDef ADV/RET + ResetReset 자동
  - { system: Cyl2, kind: passive, device: cylinder }
```

⚠ S301 같은 명칭은 *한 station 의 이름* — Flow 명이 아닙니다 (§1 매핑표). 사양에서 *여러 S* 가 등장하면 (예: S301, S302, S303 이 part 흐름으로 연결) 같은 Flow 안의 여러 Work 로 펼치고 §3.7 capacity cycle 패턴 적용.

ADV 만 사양에 있을 경우 — **§2 룰 D (추측 금지) 우선**: 기본은 ADV 만 만들고, 사용자에게 1회 "opposing RET 도 함께 만들까요? (CYLINDER 의 표준 쌍은 §3.3 표 참조)" 명확화 후 사용자 confirm 시에만 동반 생성. 임의 동반 생성은 룰 D 위반입니다.

⚠ **Passive cascade ≠ Active Call 발행** (혼동 주의): `device: cylinder` sugar 는 Passive 안에 ADV/RET 두 ApiDef 를 *항상* 만들지만 (sugar 의 정의 — 룰 D 와 무관), 그 ApiDef 를 가리키는 *Active Call* (예: `calls: [Cyl1.RET]`) 의 발행은 사용자가 RET 를 명시한 경우에만. 즉 *Passive 의 ApiDef 존재* 와 *Active calls 의 호출 발행* 은 별개 layer. ADV 만 사양에 있는 사용자에게는 `{system: Cyl1, kind: passive, device: cylinder}` (sugar 의 ADV/RET 자동 cascade) + Active 의 `calls: [Cyl1.ADV]` 만 발행이 정합.

### 4.3 "Cyl1, Cyl2 순차 ADV 후 Cyl1, Cyl2 순차 RET" — 동일 디바이스 왕복은 한 Work

올바른 분해 — 한 Work 안의 4 Call:

```yaml
protocol: promaker/v0
project: M1

systems:
  - system: Controller
    kind: active
    flow Run:
      works:
        Cycle:
          calls: [Cyl1.ADV, Cyl2.ADV, Cyl1.RET, Cyl2.RET]
          arrows:
            - Cyl1.ADV -> Cyl2.ADV : Start
            - Cyl2.ADV -> Cyl1.RET : Start
            - Cyl1.RET -> Cyl2.RET : Start
        Cycle_Clear: { calls: [] }     # SelfClear dummy (empty — Cycle 이 1-원소 cluster 의 유일 leaf)
      arrows:
        - Cycle -> Cycle_Clear : StartReset    # SelfClear forward Start + backward Reset
        - Cycle -> Cycle_Clear : Reset         # SelfClear cycle 폐쇄

  - { system: Cyl1, kind: passive, device: cylinder }
  - { system: Cyl2, kind: passive, device: cylinder }
```

⚠ **잘못된 분해 — Work 두 개로 분리 (Work1=ADV 들 / Work2=RET 들, Work1 -> Work2)** 는 *Work capacity 충돌* 로 직결됩니다:

- Work 단위는 *part 가 점유하는 페이즈* 단위 (§3.4 의 "Work 단위 = station 단위").
- 한 part 가 Work1 의 ADV 들을 끝내고 Work2 의 RET 로 넘어간 *동안*, Cyl1 / Cyl2 는 ADV 상태를 *유지한 채* Work2 에 머뭅니다.
- 동시에 Work1 은 *다음 part* 를 받아 새 ADV 를 수행해야 하지만 — 같은 Cyl1 / Cyl2 인스턴스가 이미 ADV 상태이므로 ADV→ADV 의 자기 이행이 불가 (CYLINDER 의 ResetReset opposing 시맨틱과 충돌, §3.3a (1)).
- 결과: 두 part 가 동일 디바이스 상태를 두고 충돌 → 라인 정지 또는 의미 불일치.

→ 일반화: **같은 디바이스 인스턴스가 한 part 안에서 *왕복* (ADV↔RET, UP↔DN, OPEN↔CLOSE 등) 하는 경우, opposing 두 행위를 *같은 Work 안* 의 Call 로 묶으십시오.** Work 분리는 *서로 다른 디바이스 set* 이 part 를 흘려보내는 *순차 work cluster* (§3.7 — 인접 station 마다 별개 Work) 의 경우에만.

## §5 self-check (`apply_model_doc` 발행 직전)

doc 의 JSON object 를 보내기 전 머릿속으로 다음을 확인. 하나라도 NO 면 doc 을 수정한 뒤 보냅니다.

- [ ] 최상단에 `protocol: promaker/v0` 이 있는가? (누락 시 거부)
- [ ] 모든 systems entry 에 `kind: active | passive` 가 명시되어 있는가? (추론 없음)
- [ ] 사양에 *새로* 등장한 **모든 디바이스 인스턴스**가 `systems:` 의 passive entry 로 등록되어 있는가? (`device:` 키 한 줄이 ApiDef + Flow + Work + opposing Arrow cascade 를 자동 생성)
- [ ] **예외**: 사양이 *이미 store 에 존재* 하는 PassiveSystem 만 지칭 (예: 기존 Cyl1 에 새 Active Work 만 추가) 하면 doc 의 `systems:` 에 그 entry 를 *다시* 두지 *않고* — Active calls 의 `<system>.<apidef>` dotted-path 만 사용. store snapshot / `export_model_doc` 로 기존 인스턴스 우선 확인.
- [ ] **Sibling Uniqueness 일반화** — 모든 entity 종류 (Project 하의 System / System 하의 Flow / Flow 하의 Work / Passive 하의 ApiDef) 가 *같은 부모 안 sibling 이름 unique*. store 에 기존 sibling 이 있으면 doc 의 같은 이름 entry 를 *제거* (Passive 재사용 패턴과 동일). 그렇지 않으면 MCP `queueAdd*` 단에서 fail. Call 만 예외 (같은 Work 안 같은 dotted-path call N회 = concurrent 의미, SSOT yaml-protocol-v0.md §1.7).
- [ ] **Active System 특칙** — store 에 이미 active 가 있고 사용자가 *명시적 재구성을 요청 안 한* 경우: doc 의 `- system: <X>, kind: active` 의 `<X>` 를 store 의 active 이름 그대로 사용 (dispatcher 가 silent reuse). 다른 이름 active 발행은 PLC 1 controller 규약 위반으로 MCP fail. 교체 의도면 patch 의 `rename:` / `remove:` 사용. $3.tooling.md 의 "Sibling Uniqueness — MCP 자동 검사" 표 참조.
- [ ] 모든 Active `calls: [<sys>.<api>]` dotted-path 가 같은 doc 안 (또는 store 의) Passive entry 의 ApiDef 를 정확히 가리키는가? `<sys>` 가 GUI alias 가 아니라 **Passive System 이름** 인가?
- [ ] 디바이스 행위 (ADV, OPEN 등) 가 Active 가 아니라 **Passive 의 ApiDef** 로 들어갔는가? (룰 C — Passive entry 의 `device:` sugar 또는 `apis:` 키)
- [ ] Passive entry 의 `system:` 이름이 사양의 디바이스 인스턴스 이름 (Cyl1, RB1 등) 과 정확히 일치하는가? (대소문자/공백/언더스코어 — Call dotted-path 정합성)
- [ ] **device sugar 3종 한정** — `cylinder`/`clamp`/`robot` 만 short-form. 그 외 (pusher / conveyor / agv / gripper / lifter / crane / gate 등) 는 `device: custom(<Type>), apis: [...]` long-form?
