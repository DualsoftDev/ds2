# 모델링 도메인 룰 — 사양 → Ds2 모델 분해

이 문서는 사용자가 자연어 사양 (예: "Cyl 을 전진 후 후진", "S301 에 실린더 2 개와 컨베이어 1 개", "로봇 2 대를 home → work1 → work2 순서로") 을 줄 때 그것을 **올바른 Project / DsSystem(Active|Passive) / Flow / Work / Call / ApiDef / Arrow 묶음** 으로 분해하는 의사결정 룰입니다. 3.tooling.md 가 "도구를 어떻게 호출하는가" 라면 이 문서는 "무엇을 만들어야 하는가" 입니다.

---

## §0 해석 단계 — 자연어 → Ds2 entity mapping

**전제**: 사용자는 Ds2 의 entity 어휘 (Project / DsSystem / Flow / Work / Call / ApiDef / Arrow) 를 *모를 가능성이 높습니다*. 사용자는 "공정", "라인", "스테이션", "디바이스 동작" 같은 *현장 어휘* 만 사용. 이 비대칭을 무시하고 사용자 어휘를 entity 에 *직접 1:1 매핑* 하면 길을 잃습니다 (예: 한국어 "공정" 을 자동 Flow 로 매핑 → §3.4a 의 (1) 케이스를 (2) 로 오인 → 룰 E 위반).

### 0.1 mapping 절차 (LLM 내부 수행)

1. 사양에서 *명사* / *동사* / *흐름 단서* 추출:
   - 명사 — 디바이스 인스턴스 (실린더, 클램프, 로봇, gun, spray 등), 라인 / 공정 / 스테이션 명, 컨트롤러 명.
   - 동사 — CLAMP / OPEN / SPOT / SPRAY 등 디바이스 동작.
   - 흐름 단서 — "후 / 다음 / then / after", "넘기면 / 이동", "동시 / 병렬", "다음 부품 바로", "한 station 에서 다음으로", "반복".
2. 각 명사 / 동사에 §1 매핑표 적용 → 잠정적 Ds2 entity 결정. 모호한 어휘 ("공정" 등) 는 §3.4a 결정 트리로 분기.
3. 흐름 단서에서 §3.4b / §3.6 / §3.7 의 arrow 패턴 결정 — capacity cycle vs Start-only vs ResetReset 등.
4. 절대 룰 (§2 A / B / C / D / E) 위반 검사. 위반 시 분해 단위 수정.
5. self-check (§5) 통과 후 apply_operations 발행.

### 0.2 사용자와의 대화 — *역할 어휘로 풀어쓰기*

명확화 질문 / 결과 요약 시 **DS entity 어휘 (Flow / Work / ArrowBetweenCalls / ApiDef / ResetReset Arrow 등) 를 직접 노출하지 마십시오** — 사용자는 그 어휘의 의미를 모를 가능성이 큽니다. 대신 *역할 어휘* 로 풀어 설명:

| 지양 — DS 어휘 직접 노출 | 권장 — 역할 어휘로 풀어쓰기 |
|---|---|
| "Flow `Process1` 안의 Work 3 개" | "공정 1 안의 station 3 개" |
| "ArrowBetweenWorks Start" | "한 station 끝나면 다음 station 시작" |
| "ResetReset Arrow" | "두 동작 중 하나만 활성 (상호 배타)" |
| "Passive System 의 ApiDef" | "디바이스가 제공하는 동작 (CLAMP / OPEN 등)" |
| "Work 안의 Call DAG" | "한 station 안의 sub-action 들 (순차 처리)" |
| "Self Reset arrow" | "한 단계 끝나는 즉시 자기 자신을 ready 로 복귀 (다음 부품 받을 준비)" |
| "capacity-2 cycle 의 4-arrow 패턴" | "두 station 동시 진행으로 흐름 라인 구성" |

이름 / 식별자 (Project 명, System 명, Flow 명, Work 명) 는 사용자가 *원하는 이름* 그대로 보존 — 단 *entity 종류 어휘* 만 풀어쓰기. 사용자가 "공정 명을 무엇으로 할까요?" 같이 물어오면 그대로 받아 적용.

### 0.3 결과 확인 형식 권장

분해 결과를 사용자에게 보여줄 때 — DS 어휘 트리 (`Flow → Work → Call`) 가 아니라 *역할 트리* 로:

```
공정 1 (Spot)
  station 안 sub-action: Clamp1 클램프 → Clamp2 클램프 → Gun spot → Clamp1 언클램프 → Clamp2 언클램프
공정 2 (Spray)
  station 안 sub-action: Clamp3 클램프 → Clamp4 클램프 → Spray.SPRAY → ...

흐름:
  공정 1 끝나면 → 공정 2 시작 (part 가 다음 station 으로 이동)
  공정 2 시작 시 → 공정 1 은 ready 로 복귀 (다음 부품 받을 준비)   ← 동시 진행 = capacity 2
  공정 2 끝 시 → 공정 2 는 즉시 ready 로 복귀 (다음 부품 대기)
```

내부적으로 DS 어휘 (Flow / Work / Call / Arrow) 로 매핑된 entity 가 있지만 사용자에게는 위 형태가 자연. 사용자가 *"DS 모델 어떻게 만들어졌어?"* 처럼 명시적으로 entity 어휘를 요청하면 그때 DS 어휘로 노출.

## §1 핵심 매핑표 — 현장 개념 ↔ Ds2 모델

이 표가 본 문서의 단일 진실 원천입니다. 사양에 등장하는 어휘를 좌측에서 찾아 우측의 entity 로 옮기십시오.

| 현장 어휘 | Ds2 entity | 비고 |
|---|---|---|
| 프로젝트 / 라인 / 현장명 | **Project** | 1 개. 사용자가 이름을 안 주면 1회 명확화 |
| 컨트롤러 (PLC / 호출 주체) | **Active DsSystem** (1 개) | 모든 Flow 와 ArrowBetweenWorks 가 여기 안에 |
| 공정 / 스테이션 / 라인 단계 — *part 가 머무는 한 station* (Spot 공정, Spray 공정, S301 등) | Active 의 **Work** | 한 station = part 가 통과하는 한 페이즈. 두 station 사이에 part 가 흘러가면 *같은 Flow 안의 두 Work* + capacity cycle arrow 패턴 (§3.4a / §3.7). ⚠ "공정" 어휘가 자동으로 Flow 가 *아님* — 분해 입자 결정은 §3.4a |
| 독립 트리거되는 흐름 그룹 (별개 라인 / 메인 운전 vs 알람 처리 등) | Active 의 **Flow** | 한 Flow = 단일 트리거 / 단일 part 흐름. 다른 Flow 와는 arrow 로 연결하지 않음 (룰 E). 한 Flow 에 최소 1 Work |
| 한 station 안의 sub-action ("두 clamp 순차 클램프", "spot 후 unclamp", "전진 페이즈") | Work 안의 **Call** + **ArrowBetweenCalls** (DAG) | 한 station 의 모든 sub-action 은 같은 Work 안의 Call 들이며 ArrowBetweenCalls (Start) 로 race-free 순차. 페이즈마다 Work 를 새로 만들지 *않음* |
| 한 단계에서 디바이스에 내리는 명령 ("Cyl.ADV") | Work 안의 **Call** (`{devicesAlias}.{apiName}`) | devicesAlias 는 반드시 **존재하는 Passive System 이름** |
| **물리 디바이스 인스턴스** (Cyl1, RB1, Conveyor3, MainGate) | **Passive DsSystem** (인스턴스마다 1 개) | ⚠ 누락 금지 — §2 절대 룰 |
| 디바이스의 행위 (ADV, RET, OPEN, CLOSE, HOME, WORK) | Passive 안의 **ApiDef** | 행위마다 1 ApiDef. 같은 행위가 여러 인스턴스에 공통이면 인스턴스 별로 동일 이름 ApiDef 반복 |
| 디바이스 행위의 실행 단위 | Passive 안의 **Flow → Work** (행위마다 1 Work) | GUI canonical (WithCyl.json) 형태. ApiDef.TxGuid/RxGuid = 그 행위 Work.Id binding. **현 phase MCP 도구 한계로 직접 생성 불가** — `add_api_def` 가 TxWorkId/RxWorkId 미노출. 후속 phase 의 device-class helper (`add_cylinder` 등) 가 1 op 로 자동 cascade. 현 phase 에서는 Passive 안 Flow/Work 를 *생략* 하고 Active 측만 작성 (§4.1 minimal 참조) |
| 같은 디바이스의 상반 행위 (ADV ↔ RET, UP ↔ DN, OPEN ↔ CLOSE) | **ResetReset Arrow** | Passive 의 두 Work 사이. §3.5 |
| 한 단계가 끝나면 다음 단계 ("전진 후 후진", "A 다음 B") | **Start Arrow** | Active 의 두 Work 사이 또는 Work 안의 두 Call 사이. §3.4 |
| 상호 배타 / interlock ("둘 중 하나만") | **ResetReset Arrow** | |
| 선행 시작 시 후속 abort | **Reset Arrow** | |

⚠ 잘 틀리는 부분: "Cyl 전진 후 후진" 같은 사양은 **Active 1 + Passive(=Cyl) 1 + ApiDef 2 (ADV, RET) + Active Flow 1 + Active Work 2 (또는 1 Work 2 Call) + Arrow 1** 묶음입니다. Active 만 만들고 Passive(Cyl) 를 생략하면 Call 의 `devicesAlias` 가 가리킬 대상이 없어 모델이 비정상입니다.

## §2 절대 룰 (위반하면 모델 자체가 깨짐)

### 룰 A — 디바이스가 등장하면 반드시 Passive System 을 먼저 만든다

사양에 등장하는 모든 물리 디바이스 (실린더, 모터, 컨베이어, 로봇, 밸브, 게이트, 스토퍼, 클램프, 리프터, 솔레노이드 …) 는 **인스턴스마다 1 개의 Passive DsSystem** 으로 등록합니다. Call 만 만들어 두면 안 됩니다.

잘못된 예 — Cyl 의 device(=Passive) 를 만들지 않고 Call 만 작성:
```json
[
  {"op":"add_project", "ref":"p", "args":{"name":"M1"}},
  {"op":"add_system",  "ref":"ctl", "args":{"name":"Controller", "isActive":true}},
  {"op":"add_flow",    "ref":"f",  "args":{"name":"Run", "systemId":"@ctl"}},
  {"op":"add_work",    "ref":"w",  "args":{"localName":"Cycle", "flowId":"@f"}},
  {"op":"add_call",    "args":{"devicesAlias":"Cyl", "apiName":"ADV", "workId":"@w"}}
]
```
→ devicesAlias `"Cyl"` 가 가리킬 Passive System 이 없습니다. 이 모델은 validate 에서 Orphan/DanglingArrow 류로 잡히거나, 잡히지 않더라도 의미상 비어 있습니다.

올바른 예 — Passive(Cyl) + ApiDef 를 먼저 만들고 Active 의 Call 이 그것을 참조:
```json
[
  {"op":"add_project", "ref":"p",   "args":{"name":"M1"}},
  {"op":"add_system",  "ref":"cyl", "args":{"name":"Cyl", "isActive":false}},
  {"op":"add_api_def", "ref":"adv", "args":{"name":"ADV", "systemId":"@cyl"}},
  {"op":"add_api_def", "ref":"ret", "args":{"name":"RET", "systemId":"@cyl"}},
  {"op":"add_system",  "ref":"ctl", "args":{"name":"Controller", "isActive":true}},
  {"op":"add_flow",    "ref":"run", "args":{"name":"Run", "systemId":"@ctl"}},
  {"op":"add_work",    "ref":"adv", "args":{"localName":"Adv", "flowId":"@run"}},
  {"op":"add_work",    "ref":"ret", "args":{"localName":"Ret", "flowId":"@run"}},
  {"op":"add_call",    "args":{"devicesAlias":"Cyl", "apiName":"ADV", "workId":"@adv"}},
  {"op":"add_call",    "args":{"devicesAlias":"Cyl", "apiName":"RET", "workId":"@ret"}},
  {"op":"add_arrow",   "args":{"sourceId":"@adv", "targetId":"@ret", "arrowType":"Start"}}
]
```

### 룰 B — Call 의 `devicesAlias` 는 반드시 직전에 만든 Passive System 이름과 정확히 일치

대소문자, 공백, 언더스코어까지. 사양의 "Cyl1" 을 Passive 로 만들어 놓고 Call 의 devicesAlias 를 "CYL1" 으로 쓰면 같은 디바이스로 묶이지 않습니다.

### 룰 C — ApiDef 는 Passive 안에 둔다 (Active 가 아님)

`ADV` / `RET` / `OPEN` / `CLOSE` / `HOME` / `WORK` 같은 디바이스 행위 시그니처는 그 디바이스의 Passive System 에 ApiDef 로 등록합니다. Active System 에는 ApiDef 를 만들지 않습니다 (Active 는 Flow/Work/Call 만 갖습니다).

같은 행위 이름을 여러 인스턴스가 공유하더라도 (Cyl1, Cyl2 모두 ADV/RET) Passive 마다 동일 이름의 ApiDef 를 반복 생성합니다. ApiDef 공유 메커니즘은 없습니다.

### 룰 D — 추측 금지 (3.tooling.md 의 anti-hallucination 재확인)

사양에 없으면 만들지 마십시오. "Cyl 전진 후 후진" 만 적혀 있으면:
- 만들 것: Cyl 1 개, ApiDef 2 개 (ADV, RET), Work 2 개, Arrow 1 개 (ADV → RET, Start)
- 만들지 말 것: 임의로 "초기 위치", "2차 후진", "센서 확인" 같은 단계를 끼워 넣지 않음
- 만들지 말 것: Cyl1, Cyl2 처럼 인스턴스를 임의로 늘리지 않음
- 만들지 말 것: 같은 turn 에 Conveyor / Stopper 같은 본문에 없는 디바이스를 동반 생성하지 않음

모호하면 1회 명확화 질문 후 진행 (3.tooling.md 의 Clarification 템플릿).

### 룰 E — ArrowBetweenWorks 는 같은 Flow 안의 Work 사이로만

ArrowBetweenWorks 의 source/target 은 **같은 Flow 안의 Work** 로 한정합니다 (도메인 규약). ds-entities 정의상 같은 System 안 임의 Work 사이가 *기술적으로* 가능하지만, **Flow 의 경계를 넘는 work-arrow 는 의미적으로 부자연** — Flow 가 *논리 그룹* 의 의미를 잃습니다.

→ 두 station 사이에 part 흐름이 있어 work-arrow 가 필요하다면 두 station 을 *같은 Flow 안의 두 Work* 로 만드십시오 (§3.4a 의 (1) 케이스). 두 Flow 로 분해해 놓고 cross-flow arrow 로 잇는 모델은 룰 E 위반 — 본 문서 도메인 룰의 위반.

자연스러운 다중 Flow 시나리오 (룰 E 의 비위반 케이스) 는 §3.4a 의 (2) — *독립 트리거되는 별개 흐름* (메인 운전 vs 알람 처리, 동시 가동되는 두 라인 등). 이 경우 두 Flow 는 *서로 arrow 없음* 이 정상.

## §3 분해 절차 (사양 → 모델)

### 3.1 Project

사용자가 이름을 명시하지 않으면 1회 묻고 진행. `list_projects` 가 비어 있고 사용자가 이름을 주면 같은 turn 에 add_project 1회 허용.

### 3.2 Active vs Passive 식별

사양 어휘를 두 그룹으로 분류:

| 분류 | 키워드 / 단서 | 결과 |
|---|---|---|
| **Active** (1 개) | 컨트롤러, PLC, 시퀀서, "전체 동작", "라인 제어" | Active DsSystem 1 개. 이름은 사용자 의도 — 없으면 "Controller" / "Main" / 라인 명을 묻기 |
| **Passive** (N 개) | 디바이스 명사 (실린더 / 모터 / 컨베이어 / 로봇 / 밸브 / 게이트 / 스토퍼 / 클램프 / 리프터) — 인스턴스 단위 | 인스턴스마다 1 개 Passive DsSystem |

같은 deviceType 의 인스턴스가 여러 개 (Cyl1, Cyl2, Cyl3) 면 **인스턴스마다 별도 Passive** — 묶지 않습니다.

### 3.3 디바이스 타입 → 행위 (ApiDef) 표준 매핑

해당 인스턴스가 속한 deviceType 의 표준 행위가 곧 그 Passive 의 ApiDef 들입니다.

| deviceType | 표준 ApiDef (= works) | opposing 쌍 |
|---|---|---|
| CYLINDER | ADV, RET | (ADV, RET) |
| STOPPER | ADV, RET | (ADV, RET) |
| CLAMP | CLP, UNCLP | (CLP, UNCLP) |
| LIFTER | UP, DOWN | (UP, DOWN) |
| GATE / DOOR | OPEN, CLOSE | (OPEN, CLOSE) |
| VALVE | OPEN, CLOSE | (OPEN, CLOSE) |
| MOTOR | ON, OFF *또는* CW, CCW | (ON, OFF) *또는* (CW, CCW) |
| CONVEYOR | MOVE *또는* FWD, REV | 단일 MOVE 일 때 opposing 없음 |
| PUMP | ON, OFF | (ON, OFF) |
| ROBOT | HOME, WORK *또는* HOME, WORK1, WORK2, … | opposing 없음 — HOME 은 일반적으로 입력만 |
| SOLENOID | ON, OFF | (ON, OFF) |

표에 없는 디바이스이거나 사용자가 변형 행위를 명시하면 (예: "실린더가 전진/후진/홀드 3 단") 그대로 따릅니다 — 표를 강제하지 마십시오.

ROBOT 의 차수 (WORK1, WORK2, …) — 사용자가 "로봇이 N 차 작업" 이라고 명시하면 ApiDef 를 `WORK1` … `WORK{N}` 으로 펼치고, 아니면 단일 `WORK` 만.

### 3.4 Active Flow / Work / Call 구성 — 순차 흐름

Flow 는 *독립 트리거되는 흐름 그룹* 의 단위입니다 (§1 매핑표 / §3.4a). 사양에 그런 그룹 분리 단서가 없으면 (예: "Cyl 전진 후 후진" 만) Flow 1 개 ("Run" / "Cycle" 등) 로 두십시오 — 사용자가 이름을 안 주면 묻습니다.

한 Work = 한 *station* (= part 가 통과하는 한 페이즈). station 안의 모든 sub-action (CLAMP × N → 메인 동작 → UNCLAMP × N 같은 페이즈 내 모든 동작) 은 *같은 Work 안* 의 Call 들로 묶고 ArrowBetweenCalls (Start) 로 race-free 순차. **페이즈마다 Work 를 새로 만들지 마십시오** — Work 단위는 *station 단위* 이지 *동작 단위* 가 아닙니다.

순차성 ("후", "다음", "then", "after") → 같은 Work 안 두 Call 사이라면 **ArrowBetweenCalls** (Start). 두 station (=두 Work) 사이라면 **ArrowBetweenWorks** (Start), 단 capacity / part-flow 단서가 함께 등장하면 §3.4b / §3.7 의 capacity cycle 패턴 (Start + Reset + Self Reset) 으로 갑니다.

### 3.4a Flow vs Work — 분해 입자 결정 트리

사용자 사양에 "공정", "단계", "스테이션", "두 라인" 처럼 다단계 언급이 있을 때 — **Flow 로 분해할지 Work 로 분해할지** 결정하는 룰. 한국어 "공정" 이 자동으로 Flow 가 아닙니다.

(1) 두 단계 사이에 *part / 토큰 / 자재가 흘러간다* (예: "spot 공정 끝나면 spray 로 넘기면", "S301 끝나면 다음 station 으로", "1차 공정에서 다음 공정으로 part 가 이동") → **같은 Flow 안의 두 Work**.
- capacity > 1 cycle 의 신호가 함께 등장하면 (§3.4b) §3.7 의 4-arrow capacity cycle 패턴 적용.
- Flow 사이 arrow 를 만들지 *않습니다* (룰 E).

(2) 두 단계가 *독립적으로 트리거되는 별개 흐름* (예: 메인 운전 vs 알람 처리, 동시 가동되는 두 라인, 모드별 선택 실행) → **두 Flow**.
- 두 Flow 사이는 arrow 없음 (룰 E). 각 Flow 가 독립 cycle.
- 각 Flow 가 자체 외부 트리거 (button / 알람 신호 / 상위 mode 신호 등) 로 시작.
- ⚠ 두 흐름 간 *신호 의존* 이 있다면 (예: "알람 발생 시 메인 Flow 정지") — 룰 E 강제 하 다음 두 옵션 중 택일: **(a) 같은 Flow 로 합치기** (단일 Flow 안 두 work-cluster + 가운데 Reset arrow 또는 StartReset arrow 로 의존 표현), 또는 **(b) 후속 phase 의 HwCondition / HwAction 으로 외부화** (현 phase 도구 외 — IO 신호 매개로 의존 표현. `3.tooling.md` 의 "본 phase 의 도구 범위" 참조). 현 phase 에서는 (a) 가 보통 정합.

(3) 두 단계가 *순차이지만 part 흐름이 아닌 단순 sequencing* (예: 시스템 초기화 후 운전, 충돌 검사 후 출하) → **같은 Flow 안의 두 Work, A --> B (Start) 만**.
- Reset / Self Reset 없음. capacity = 1 (한 cycle 1회 처리 후 종료 또는 외부 trigger 대기).

⚠ **잘못된 분해의 신호**: 사용자가 "두 공정" / "1차, 2차 공정" 이라 했다고 자동으로 두 Flow 로 가지 마십시오. (1) / (2) / (3) 중 어디에 해당하는지 *흐름 단서* 로 판별. 모호하면 (1) 을 default 로 제시하고 1회 명확화 — 자동차 / 가공 라인 도메인은 (1) 이 압도적 다수.

### 3.4b capacity / 파이프라인 어휘 인식

다음 어휘 / 단서가 사양에 등장하면 §3.7 의 capacity-N cycle 패턴 (Start + Reset + Self Reset 4-arrow) 으로 모델링:

| 사양 어휘 / 단서 | 패턴 |
|---|---|
| "동시 진행" / "동시에 처리" / "두 단계 동시" / "병렬 가공" | capacity-N cycle |
| "다음 부품이 *바로* 들어올 수 있다" / "다음 part 가 곧바로" / "흐름 라인" / "파이프라인" / "처리량" | capacity-N cycle |
| "spot 한 다음 spray 로 *넘기면* 다음 부품 spot 가능" 같이 *다음 단계로 넘기는 묘사* | capacity-N cycle |
| 단순 "후" / "다음" / "then" / "after" *만* 있고 part 흐름 / 병렬 단서 없음 | §3.6 의 Start arrow only (capacity 1) |

§3.4a 의 (1) 케이스 (part 흐름) 와 결합하면 거의 항상 capacity > 1 cycle 패턴. 어휘가 명확치 않으면 사용자에게 "다음 부품이 첫 station 에 *바로* 들어올 수 있는 흐름 라인입니까, 아니면 한 부품 1회 처리로 충분합니까?" 1회 명확화.

### 3.5 Passive 내부 — opposing Arrow (후속 phase 의 helper 영역)

같은 디바이스의 상반 행위 (CYLINDER 의 ADV ↔ RET 등) 는 Passive 안에서 **ResetReset Arrow** 로 연결합니다. 이는 "둘 중 하나만 활성" 이라는 상호 배타 시맨틱.

작성 위치: 같은 Passive 안에 행위마다 Flow → Work 가 있으면 그 두 Work 사이.

opposing 이 없는 디바이스 (CONVEYOR.MOVE 단일, ROBOT.WORK 단일) 에는 ResetReset 을 넣지 마십시오.

⚠ **현 phase 의 MCP 도구 한계**: 본 절의 Passive 내부 Flow → Work + ResetReset Arrow + ApiDef.TxGuid/RxGuid binding 은 GUI canonical (WithCyl.json) 의 형태이지만, 현 phase 의 primitive 도구 (`add_api_def(name, systemId)`) 가 TxWorkId/RxWorkId 인자를 노출하지 않아 **현 phase 에서는 직접 생성하지 않습니다**. 후속 phase 의 device-class helper 4종 (`add_cylinder` / `add_clamp` / `add_robot` / `add_device` — `todo-extend-mcp.md` 참조) 이 1 op 로 PassiveSystem + Flow + Work + ApiDef + ResetReset Arrow cascade 를 자동 생성하도록 설계되어 있으며, 그 시점에 본 절의 권고가 발효됩니다. 현 phase 의 정합 모델은 §4.1 의 Active 측 minimal 묶음으로 한정.

### 3.6 Arrow 시맨틱 결정 트리

사양 어휘로부터 arrowType 을 결정 (3.tooling.md 의 Arrow 시맨틱 절 보강):

| 사양 어휘 | arrowType |
|---|---|
| "후", "다음", "then", "after", "→" | Start |
| "선행이 시작되면 후속은 abort" | Reset |
| "한쪽이 시작되면 다른 쪽 reset (양방향)", "interlock" | StartReset |
| "둘 중 하나만", "동시에 안 됨", **opposing 쌍** | ResetReset |
| 단순 그룹화 / UI 클러스터링 | Group |
| 시맨틱 불명 — 묻기 전 사용 금지 | Unspecified |

### 3.7 실행 시맨틱 — Arrow 신호 / Self Reset / capacity cycle

본 절은 *모델링 의사결정의 근거* 입니다 — Arrow 의 타입 결정, capacity cycle 패턴 인식이 모두 본 시맨틱에 기반.

#### 표기 약속 (본 절 내, line style 로 신호 종류 구분)
- `-->` = Start arrow (solid line — 순간 신호)
- `..>` = Reset arrow (dotted line — 지속 신호)
- `.->` = StartReset arrow (dash-dotted — Start + Reset 결합)

> 본 문서의 화살표 표기는 두 layer 입니다 — 혼동하지 마십시오.
> - **자연어 layer** (사용자가 사양에 쓰는 어휘): `→` (unicode arrow). §3.6 의 "후 / 다음 / then / after / →" 어휘 매핑표 등.
> - **DSL layer** (본 문서의 표기 약속): `-->` / `..>` / `.->` (ASCII, line style 로 신호 종류 구분). §3.7 / §3.4a 의 모델링 다이어그램 등.
>
> 두 layer 가 동일 의미 (예: Start) 일 때도 *표기는 다름*. 자연어 layer 의 `→` 를 DSL 표기로 자동 매핑하지 마십시오 — 사용자 어휘를 §3.6 / §3.4b 결정 트리로 해석한 결과가 DSL 표기.

#### work 의 RGFH FSM
- work 상태: ready → going → finish → homing → (다시) ready 의 4-상태 cycle.
- going 상태에서 같은 work 의 새 going 시작 불가 (single instance 보장). 재실행 = finish 후 reset 신호 → homing → ready 도달 후.

#### Arrow 신호 시맨틱 (RGFH 와의 연결)
하나의 ArrowBetweenWorks 는 *두 종류 신호* 를 동시에 내포 (이중성):
- **Start arrow** (`-->`): source 의 Finish *rising edge* (순간 신호) → target 을 ready 에서 going 으로 트리거.
- **Reset arrow** (`..>`): source 의 Going *high level* (지속 신호) → target 이 finish 라면 homing 으로 전이시킴.
- **StartReset arrow** (`.->`): 위 두 신호의 union — A 끝나면 B 시작 + A 진행 중인 동안 B 는 reset 상태 유지 (인터록).

Start = 순간 신호, Reset = 지속 신호. **이 비대칭이 cyclic flow 의 동작 원리**이며, capacity > 1 cycle 의 핵심 메커니즘.

본 절은 *Work-arrow 의 Start / Reset / StartReset 신호 시맨틱* 만 다룹니다. **ResetReset** 은 §3.6 결정 트리 / §3.5 의 Passive 내부 opposing 쌍 (CYLINDER ADV ↔ RET 등) 영역으로 — 본 절의 Work-arrow 시맨틱과 다른 layer (디바이스 내부 상호 배타). **Group** 은 UI 클러스터링 hint 라 실행 시맨틱 없음 (§3.6).

#### work 안의 call DAG
- work 가 going 상태가 되면 그 안 call 들이 ArrowBetweenCalls (DAG) 따라 순차 진행.
- call 도 개념적으로 RGFH 이지만 외부 시스템 호출이라 going / homing 미관찰 — ready 에서 ApiCall 발행 → 센서 감지 시 직접 finish.
- DAG race-free: 같은 work 안 두 call 이 *동시에 going* 일 수 없음.
- e.g: work 안에 call1, call2, call3 이 있고 `call1 --> call2 --> call3` ArrowBetweenCalls 가 있다면 work going 시 call1 → call2 → call3 순차 실행. call2 진행 중 call1 / call3 실행 불가.

#### Self Reset 패턴 (`work ..> 자기 자신`)
- 의미: work 가 종료(finish) 도달 *즉시* 자기 자신을 reset → finish → homing → ready 즉시 복귀.
- 효과: 외부 reset 신호 없이 work 가 자율적으로 cycle 을 닫음 — 반복 작업 work 의 표준 closing.

#### capacity-2 cycle 의 표준 4-arrow 패턴

두 station (workA, workB) 가 part 를 흘려 보내며 *동시 진행 가능* 한 cycle 의 표준 모델:

```
외부신호 (button / 도착 센서 등) --> workA   (Start)
workA --> workB                                 (Start)
workB ..> workA                                 (Reset)
workB ..> workB                                 (Reset, Self Reset)
```

동작 단계:
1. 외부 신호 → workA going → finish (Finish↑) → workB going 트리거.
2. workB going (high level) → workA reset 신호 → workA 가 finish 였다면 homing → ready 복귀.
3. workA ready + 외부 신호 다시 → workA 재 going. 이때 workB 도 여전히 going 가능 (concurrency).
4. workB self-reset 으로 finish 즉시 homing → ready → 다음 start 대기.

→ workA / workB 동시 going 가능한 그래프 → 처리 capacity 가 2 (KPI 차원의 capacity 는 후속 phase 의 Simulation 도메인. 본 항은 *그래프 구조* 차원의 의미).

3+ station 으로 확장: 인접 station 사이 동일 4-arrow 패턴 반복 (workA-B 짝 + workB-C 짝 + ...).

## §4 표준 묶음 예시 (사양 → 1 회 apply_operations)

### 4.1 "Cyl 전진 후 후진" — 가장 단순 (현 phase minimal)

ActiveSystem(Controller) 1 + Passive(Cyl) 1 + ApiDef 2 (Passive 자식) + Active Flow 1 + Active Work 2 + Call 2 + ArrowBetweenWorks 1 (Start). §2 룰 A 의 "올바른 예" 그대로.

⚠ 본 minimal 묶음은 **Passive 내부의 Flow / Work / ResetReset Arrow + ApiDef.TxGuid/RxGuid binding 을 *생략*** 합니다 — §3.5 의 도구 한계 사유. GUI canonical 형태 (WithCyl.json) 와 비교하면 Passive 내부 cascade 가 빠져 있으며, 후속 phase 의 device-class helper (`add_cylinder` 등) 가 도입되면 그 1 op 가 자동으로 채웁니다.

대안: "전진과 후진을 한 묶음 cycle 로" 로 해석한다면 Work 1 개에 Call 2 개 + ArrowBetweenCalls 1 개 (Start). 사용자 의도가 단계 분리인지 한 cycle 인지 모호하면 1회 묻기.

### 4.2 "S301 공정에서 Cyl1, Cyl2 가 순차 전진"

§1 매핑표 적용: S301 = 한 station = Work. Cyl1.ADV / Cyl2.ADV = S301 안의 두 sub-action = 같은 Work 안의 Call DAG (§3.4 / §1 의 한 station 안 sub-action 룰).

```
Project       — M1
ActiveSystem  — Controller
  Flow        — Production                    /* 단일 흐름 — §3.4a (2) 의 별개 흐름 단서 없음 */
    Work      — S301                          /* S301 = 한 station = 한 Work */
      Calls   — Cyl1.ADV --> Cyl2.ADV         /* 한 Work 안 ArrowBetweenCalls (Start) */
PassiveSystem — Cyl1
  ApiDef      — ADV, RET                      /* 현 phase: Passive 내부 Flow/Work/ResetReset 생략 (§3.5 / §4.1 도구 한계) */
PassiveSystem — Cyl2
  ApiDef      — ADV, RET                      /* 후속 helper (`add_cylinder`) 도입 시 +Flow → Work → Work + ResetReset(ADV,RET) 자동 cascade */
```

⚠ S301 같은 명칭은 *한 station 의 이름* — Flow 명이 아닙니다 (§1 매핑표). 사양에서 *여러 S* 가 등장하면 (예: S301, S302, S303 이 part 흐름으로 연결) 같은 Flow 안의 여러 Work 로 펼치고 §3.7 capacity cycle 패턴 적용.

ADV 만 사양에 있을 경우 — **§2 룰 D (추측 금지) 우선**: 기본은 ADV 만 만들고, 사용자에게 1회 "opposing RET 도 함께 만들까요? (CYLINDER 의 표준 쌍은 §3.3 표 참조)" 명확화 후 사용자 confirm 시에만 동반 생성. 임의 동반 생성은 룰 D 위반입니다 (이전 revision 의 "자동 동반이 자연스럽다" 표기는 룰 D 와 충돌하므로 본 revision 에서 정정).

### 4.3 "로봇 RB1 이 home → work1 → work2 후 home 복귀"

```
PassiveSystem — RB1
  ApiDef      — HOME, WORK1, WORK2
ActiveSystem  — Controller
  Flow        — RobotCycle
    Work A    — Home0      Call — RB1.HOME
    Work B    — Work1      Call — RB1.WORK1
    Work C    — Work2      Call — RB1.WORK2
    Work D    — HomeReturn Call — RB1.HOME
    Arrow     — A→B→C→D   (모두 Start)
```

ROBOT 은 §3.3 표대로 opposing 없음 — ResetReset 안 만듭니다.

### 4.4 "GATE 가 OPEN/CLOSE 상호 배타, 라인 시작 시 OPEN"

```
PassiveSystem — Gate
  ApiDef      — OPEN, CLOSE       /* 현 phase: Passive 내부 cascade 생략 (§3.5 / §4.1) */
                                  /* 후속 helper (`add_device` deviceType="Unit") → +Flow + Work×2 + ResetReset(OPEN,CLOSE) */
ActiveSystem  — Controller
  Flow        — LineStart
    Work      — Open        Call — Gate.OPEN
```
