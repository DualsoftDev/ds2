# DS / EV2 Entity 모델 — Greenfield Modeling 참조

본 문서는 LLM 이 자연어 사양 → Ds2 모델 (Project / DsSystem / Flow / Work / Call / ApiDef / Arrow) 분해를 수행하는 데 필요한 **Entity 핵심 구조** 만 정리합니다.

운영 / 런타임 / AAS export / DSL / IDTA 표준 서브모델 / SubmodelProperty 32 / 이중성 8 Case 등 GFM 의사결정에 직접 기여하지 않는 영역은 본 문서에서 제외합니다 (필요 시 별도 자료 — `Solutions/Core/Ds2.Core/Entities.fs`, `Solutions/Convert/Ds2.JsonFormatter/json-format.md` 참조).

---

## 1. 핵심 설계 철학 (GFM 의사결정에 영향)

1. **그래프 기반 구조화**: Vertex(Work, Call) + Edge(Arrow) 로 흐름을 명시적으로 표현.
2. **사이클 허용 범위**:
   - Work *내부* Call 그래프: **DAG (비순환)** — `ArrowBetweenCalls`.
   - Work *간* (System 직속) 그래프: **순환 허용** — `ArrowBetweenWorks`.
3. **Active vs Passive**: 동일 `DsSystem` 타입이 Project 측 등록 위치 (`activeSystemIds[]` vs `passiveSystemIds[]`) 로 역할 분기. Active = 제어 로직 보유 / Passive = ApiDef 노출하는 디바이스.

---

## 2. Entity 트리 — GFM 핵심 골격

```text
DsStore
 ├─ Project                                              // 최상위
 │   ├─ activeSystemIds  : Guid[]                        // → DsSystem (Active)
 │   └─ passiveSystemIds : Guid[]                        // → DsSystem (Passive = Device)
 │
 ├─ DsSystem (Active 또는 Passive)
 │   ├─ iri / systemType                                 // SystemType: "Unit" | "Robot" | "Conveyor" | ...
 │   │
 │   ├─ Flow (parentId = System.Id)
 │   │   └─ Work (parentId = Flow.Id, Name = "FlowPrefix.LocalName")
 │   │       ├─ Call (parentId = Work.Id, Name = "DevicesAlias.ApiName")
 │   │       │   └─ apiCalls : ApiCall[]                 // 내장 — Call 의 ApiDef binding (도구 자동 cascade)
 │   │       └─ ArrowBetweenCalls (parentId = Work.Id)   // 내부 DAG
 │   │
 │   ├─ ApiDef (parentId = System.Id, Passive System 만 보유)
 │   │   ├─ txGuid : Guid?       // TX Work (Passive 내부 Work)
 │   │   └─ rxGuid : Guid?       // RX Work (Passive 내부 Work)
 │   │
 │   └─ ArrowBetweenWorks (parentId = System.Id)         // System 레벨 (Cyclic)
```

---

## 3. 공통 베이스 — 모든 Entity 의 식별/관계

모든 entity 는 다음 3 속성을 공유합니다 (DsEntity / DsChild / DsArrow 추상 계층):

| 속성       | 의미                                                                       |
|------------|----------------------------------------------------------------------------|
| `Id`       | 객체의 GUID (영구 식별자, import/export 기준, 항상 non-null)                 |
| `Name`     | 사람이 읽는 이름. 일반적으로 중복 허용 (단 Project 명은 unique 권장)         |
| `ParentId` | 트리에서의 부모 entity GUID (최상위 Project 만 ParentId 없음)               |

Arrow entity (`ArrowBetweenWorks` / `ArrowBetweenCalls`) 는 위 3 속성에 더해 `SourceId / TargetId / ArrowType` 을 가지며, `Name` 은 사용하지 않습니다.

## 4. Entity 상세 — GFM 도구로 만드는 영역

### 4.1 Project — 최상위 단위

| 속성                | 타입            | 설명                                                       |
|---------------------|-----------------|------------------------------------------------------------|
| `Name`              | string          | 프로젝트 명 (unique 권장)                                   |
| `ActiveSystemIds`   | `Guid[]`        | **Active System** (사용자 제어 로직, CPU 코드 생성 대상)     |
| `PassiveSystemIds`  | `Guid[]`        | **Passive System / Device** (외부 호출 대상)                |

> ⚠ **Promaker SDI 매핑**: Ds Project ↔ Promaker Project 는 **1:1**. Promaker 는 SDI (한 번에 단일 Project). 사용자가 새 Project 생성을 요청 시 — 현재 열린 Project 가 clean 이면 자동으로 닫고 새로 생성, *dirty* 이면 사용자에게 UI 로 닫아줄 것을 요청한 후 진행.

### 4.2 DsSystem — 장치·설비 단위 (Active 또는 Passive)

| 속성          | 타입             | 설명                                                                |
|---------------|------------------|---------------------------------------------------------------------|
| `Name`        | string           | System 이름 (한 Project 내 unique 권장)                              |
| `SystemType`  | `string?`        | DevicePresets.KnownNames 19종 (`Ds2.View3D.Core/Types.fs`) — `"Robot" | "Conveyor" | "Unit" | "AGV" | "Gripper" | "Lifter" | "Crane" | "Stacker" | "Sorter" | "Transfer" | "Barrier" | "Door" | "Gate" | "Elevator" | "Hoist" | "Pusher" | "Rotary" | "Turntable" | "Tilter"`. cylinder / clamp 류 simple 2-state device 는 `"Unit"`, robot 은 `"Robot"`, conveyor 는 `"Conveyor"`. 표 외 deviceType 사양 시 LLM 이 사용자 어휘 그대로 보존 (예: `"Cylinder_1"`) — 단 3D view 자동 매핑은 KnownNames 19종 한정. 디바이스 동작 메커니즘 default (복동/단동/단일 action 추론) 은 `2.modeling.md` §3.3a 참조 |

| 분류                       | 설명                                                                                 |
|----------------------------|--------------------------------------------------------------------------------------|
| **Active System**          | Project 의 `activeSystemIds` 에 등록. 사용자 제어 로직 보유. CPU 로 직접 제어.          |
| **Passive System (Device)**| Project 의 `passiveSystemIds` 에 등록. ApiDef 만 노출. 외부에서 호출받는 수동 시스템.  |

### 4.3 Flow — System 내부의 논리적 그룹

| 속성        | 타입    | 설명                              |
|-------------|---------|-----------------------------------|
| `Name`      | string  | Flow 이름 (한 System 내 unique)    |
| `ParentId`  | `Guid`  | → System.Id                       |

> Flow 자체는 Work 간 Arrow 정보를 가지지 않습니다 (`ArrowBetweenWorks` 의 ParentId 는 System).

> ⚠ **실행 의미**: Flow 는 Work 들의 *grouping container*. ds engine 이 Flow 의 Start/Stop 버튼을 자동 생성. Flow 가 Start 되면 그 안의 Work 들 중 *ready 조건을 충족하는 모두가 동시 병렬* 로 실행 (work-arrow 의 인과 제약을 따름).

### 4.4 Work — 작업 단위

| 속성          | 타입              | 설명                                              |
|---------------|-------------------|---------------------------------------------------|
| `LocalName`   | string            | Work 고유 이름 (한 Flow 내 unique)                  |
| `FlowPrefix`  | string            | 부모 Flow 이름                                     |
| `Name`        | string (자동)     | `"{FlowPrefix}.{LocalName}"` — 자동 계산           |
| `ParentId`    | `Guid`            | → Flow.Id                                          |
| `Duration`    | `TimeSpan?`       | Call 이 없을 때만 의미 (GFM 에서는 helper 가 채움)   |

### 4.5 Call — Work 내부의 API 호출 노드

| 속성             | 타입                   | 설명                                                       |
|------------------|------------------------|------------------------------------------------------------|
| `DevicesAlias`   | string                 | Device 별칭 — Passive System.Name 과 정확히 일치 강제         |
| `ApiName`        | string                 | API 이름 — Passive 의 ApiDef.Name 과 정확히 일치 강제          |
| `Name`           | string (자동)          | `"{DevicesAlias}.{ApiName}"` — 자동 계산                    |
| `ParentId`       | `Guid`                 | → Work.Id                                                  |
| `ApiCalls`       | `ApiCall[]` (내장)     | Call 의 ApiDef binding — `add_call` 도구가 자동 cascade      |

`ApiCall` 은 도구 자동 cascade 로 생성되며 LLM 이 직접 다루지 않습니다 (`apiDefId`, `originFlowId` 자동 채움).

> ⚠ **Init Call** — 한 Work 안 Call 들의 `ArrowBetweenCalls` (Start) 그래프는 **DAG (cycle 금지)**. *incoming Start arrow 가 없는 Call* 을 **init call** 이라 하며, Work 가 Start 되면 즉시 실행됨. 복수의 init call 이 가능하며 (= Work 안에 chain 이 여러 개), Work Start 시 모든 init call 이 *동시 병렬* 로 실행됨.

### 4.6 ApiDef — Passive 가 노출하는 API 인터페이스

| 속성        | 타입       | 설명                                                  |
|-------------|------------|-------------------------------------------------------|
| `Name`      | string     | API 이름 — Active 의 Call.ApiName 과 일치 강제           |
| `ParentId`  | `Guid`     | → **Passive** System.Id (Active System 자식 금지 — 룰 C) |
| `TxGuid`    | `Guid?`    | TX Work — Passive 내부 Work 와 binding (helper 가 자동 채움) |
| `RxGuid`    | `Guid?`    | RX Work — 동상                                        |

> GUI canonical (WithCyl.json) 에서는 `TxGuid = RxGuid = 그 행위의 Passive 내부 Work.Id`. **device-class helper** (`add_cylinder` / `add_clamp` / `add_robot` / `add_device`) 가 PassiveSystem cascade 시점에 자동 채웁니다. primitive `add_api_def` 만 사용하는 경우 LLM 이 `txWorkId?` / `rxWorkId?` 인자로 명시 가능.

### 4.7 Arrow — 인과 / 흐름 연결

| 속성           | 타입         | 설명                              |
|----------------|--------------|-----------------------------------|
| `SourceId`     | `Guid`       | 시작 entity (Work 또는 Call)        |
| `TargetId`     | `Guid`       | 끝 entity (Source 와 같은 종류)     |
| `ArrowType`    | enum         | 아래 표                            |
| `ParentId`     | `Guid`       | Work-arrow → System.Id / Call-arrow → Work.Id |

| 종류                | 위치 (`ParentId`) | 그래프 특성        |
|---------------------|-------------------|--------------------|
| `ArrowBetweenWorks` | System            | Cyclic Directed    |
| `ArrowBetweenCalls` | Work              | DAG (비순환)       |

#### ArrowType
| 값 | 이름           | 설명                                                                        |
|----|----------------|-----------------------------------------------------------------------------|
| 0  | `Unspecified`  | 연결 없음 — 사용자 의도 명확화 전 사용 금지                                   |
| 1  | `Start`        | source 완료 → target 시작                                                    |
| 2  | `Reset`        | source 시작 → target 리셋                                                    |
| 3  | `StartReset`   | source 완료 → target 시작 + target 시작 → source 리셋                         |
| 4  | `ResetReset`   | source 시작 → target 리셋 + target 시작 → source 리셋 (opposing 쌍 / 순환 리셋) |
| 5  | `Group`        | UI 그룹화 / 클러스터링 hint (실행 시맨틱 없음)                                  |

## 5. 빠른 참조

### 5.1 부모 관계 (`ParentId`)

| Entity              | ParentId 대상      | 비고                                                |
|---------------------|--------------------|-----------------------------------------------------|
| Flow                | System.Id          |                                                     |
| Work                | Flow.Id            | Name = `FlowPrefix.LocalName` (자동 계산)            |
| Call                | Work.Id            | Name = `DevicesAlias.ApiName` (자동 계산)            |
| ApiDef              | System.Id          | (Passive System 만)                                 |
| ApiCall             | (Call 내장)        | 별도 ParentId 없음, Call.ApiCalls 배열에 보관 (도구 자동) |
| ArrowBetweenWorks   | System.Id          | Cyclic 허용                                         |
| ArrowBetweenCalls   | Work.Id            | DAG                                                 |

### 5.2 Project 의 시스템 분류

| 분류                | 등록 위치              | 의미                                                            |
|---------------------|------------------------|-----------------------------------------------------------------|
| Active System       | `activeSystemIds[]`    | 사용자 제어 로직, CPU 코드 생성 대상                              |
| Passive System      | `passiveSystemIds[]`   | Device 역할, ApiDef 노출, 외부에서 호출됨                          |

## 6. GFM 범위 외 — 후속 phase 확장 예정

다음 entity / property 는 ds2 entity 모델에 정의되어 있으나 본 phase MCP 도구로는 직접 다루지 않습니다. 자세한 안내는 `3.tooling.md` 의 "본 phase 의 도구 범위 — 후속 확장 예정 entity" 절 참조.

- **IOTag** (PLC 물리 주소 binding — `X100`, `M300`, `DB100.DBX0.2`)
- **ValueSpec** (input/output spec, 단일/다중/범위)
- **CallCondition** (precondition tree, AND/OR, AutoAux/ComAux/SkipUnmatch)
- **TokenSpec / TokenValue** (recipe / 제품 토큰 매핑)
- **HwButton / HwLamp / HwCondition / HwAction** (Flow 의 UI 컴포넌트)
- **SubmodelProperty 32** (4 entity × 8 도메인 — Simulation / Control / Monitoring / Logging / Maintenance / Hmi / Quality / CostAnalysis)
- **ApiDefActionType** (Normal / Push / Pulse / Time(ms))
- **Work.TokenRole / ReferenceOf / Position / Status4** (런타임 / RGFH FSM)
- **Nameplate / HandoverDocumentation / TechnicalData** (IDTA 표준 서브모델)
- **AAS Semantics (ConceptDescription / Submodel idShort)** (export 시 의미)
- **DSL** (DS Language — 본 phase 는 MCP JSON op 만 사용)

위 영역의 사양이 사용자 메시지에 등장하면 *수신* 만 하고 (메모) 모델링은 현 phase 골격까지로 한정. 사용자에게 "후속 phase 예정" 1줄 안내 후 진행하십시오.
