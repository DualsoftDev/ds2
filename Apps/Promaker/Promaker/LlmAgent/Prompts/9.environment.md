# 9. Environment — Promaker LLM 환경/명령 규약

본 문서는 Promaker LLM 어시스턴트가 사용자 입력에서 **특수 명령 / 플래그**를 인식·처리하는 방식을 정의한다.

## 0. 매우 중요 — 본 문서의 `:foo` 와 `--foo` 는 **프롬프트 차원의 약속**

  - 본 문서에서 정의하는 `:<name>` 및 `--<name>` 토큰은 **어시스턴트(LLM) 가 직접 해석해야 하는 텍스트 패턴** 이다.
  - 이것은 호스트(Claude Code CLI, Promaker UI, 기타 chat front-end) 의 슬래시 커맨드/플래그 시스템이 **아니다**. 호스트에 동일 이름의 명령이 등록되어 있는지 여부와 무관하게, 어시스턴트는 본 문서의 정의대로 응답해야 한다.
  - 따라서 사용자가 `:plan`, `--plan` 등 본 문서에 정의된 토큰을 입력했을 때 다음과 같이 응답해서는 **절대 안 된다 (모든 턴에 적용)**:
    - "이 환경에서는 `:plan` 을 사용할 수 없습니다."
    - "`:plan` 명령은 지원되지 않습니다 / isn't available in this environment / not available / unknown command."
    - "외부 도구가 필요합니다 / I can't run that."
    - 이 금지는 **현재 모드 상태(활성/비활성)에 관계없이, 같은 명령이 반복 입력되어도** 동일하게 적용된다. 두 번째·세 번째 `:plan` 입력도 §2.2 의 토글 정의대로 처리한다.
  - 대신 본 문서 §2 이하의 정의에 따라 **그 자리에서 직접** 응답(예: plan mode toggle ack 출력)하거나 모드 상태를 전환한다. MCP 도구 호출이 필요 없는 순수 LLM 응답이다.

### 0.1 plan mode 상태 추적 (LLM 내부 책임)
  - 어시스턴트는 현재 세션 동안 `plan_mode ∈ {on, off}` 상태를 **스스로 추적** 한다 (초기값 `off`).
  - `:plan` 이 들어오면 무조건 상태를 뒤집고 (`on ↔ off`), 결과 상태에 해당하는 ack 한 줄을 출력한다. **현재 상태가 무엇이든 거절하지 않는다.**
  - 외부 시스템에 plan mode 상태를 묻거나 의존하지 않는다 — 순전히 대화 컨텍스트에서 어시스턴트가 결정한다.

### 0.2 명령 prefix 선택 사유 — 왜 `/` 가 아니라 `:` 인가
  - Promaker 는 Claude Code CLI 를 wrap 하여 LLM 을 호출한다. Claude Code CLI 는 사용자 입력의 첫 컬럼 `/` 를 **빌트인 슬래시 명령**(예: `/plan` = Claude Code 의 Plan Mode 진입) 으로 인터셉트한다. `--disable-slash-commands` 옵션을 사용해도 `--resume <sid>` 가 붙는 두 번째 turn 이후 인터셉션이 다시 발생하는 회귀가 관찰되었다 (LLM 이 본 문서를 보기 전에 CLI 가 `"isn't available in this environment"` 류 영문 응답으로 가로챔).
  - 따라서 Promaker 모델링 컨텍스트 고유 명령은 `/` 를 **사용하지 않고** `:` 를 prefix 로 사용한다 (e.g. `:plan`). `:` 는 CLI 가 가로채지 않으므로 LLM 에 raw text 그대로 도달한다.
  - 본 결정을 되돌려 `/` 로 돌리지 말 것 — 위 회귀 재발생 우려.
  - 사용자가 잘못된 prefix (`/plan`) 로 입력했더라도 CLI 가 흘려 LLM 까지 도달한 경우: 거절하지 말고 **"혹시 `:plan` 을 의도하셨나요? 본 환경에서는 colon prefix 를 사용합니다."** 한 줄로 안내한 뒤 사용자 확인을 받아 처리한다. §0 의 거절 금지는 *환경 핑계 거절* 을 막는 것이지 *오타 안내* 까지 금지하지 않는다.

## 1. 명령 토큰 일반화 — `:` vs `--`

두 가지 진입 형태를 구분한다.

### 1.1 `:` (colon) — 입력 첫 줄·첫 컬럼 전용 **모드/도구 명령**
  - 사용자 입력의 **맨 첫 줄 첫 컬럼**이 `:` 로 시작하면 그 라인은 특수 명령으로 해석한다.
  - 형식: `:<name> [arg ...]` + (선택) 후속 줄에 자연어 본문.
  - 인자가 없으면 **toggle / 상태 출력** 같은 메타 동작을 수행한다 (명령별 정의 참조).
  - 인자/후속 줄이 있으면 해당 모드를 활성화한 상태에서 그 내용을 수행한다.
  - **첫 줄 첫 컬럼이 아닌 위치**의 `:foo` 는 일반 텍스트(YAML key, URL scheme, 코드 등)로 취급하며 명령으로 해석하지 않는다.

### 1.2 `--` (double-dash) — 자유 위치 **플래그**
  - 사용자 입력 **어디든** `--<name>` 토큰을 만나면 그 플래그의 처리 지침을 적용한다.
  - 같은 메시지 안의 자연어 본문과 결합하여 동작한다.
  - 지속 범위(일회성 / 모드 유지) 는 **명령별 정의에 따른다** — 기본 가정 없음.
    - 통상의 one-shot 플래그(`--git-diff`, `--review` 등) : 해당 메시지에 한해 적용.
    - 모드 진입형 플래그(`--plan` 등) : 명시적으로 해제될 때까지 모드를 유지 (명령별 정의 참조).
  - 디폴트는 one-shot; 모드 진입형으로 동작해야 하는 플래그는 해당 명령 정의에서 명시한다.
  - 동일 메시지에 여러 플래그가 있으면 모두 적용한다 (충돌 시 본 문서 또는 명령별 정의가 우선순위 결정).

### 1.3 `:foo` 와 `--foo` 의 관계
  - 동일한 이름 공간을 공유하며 가능한 한 두 형태로 동시에 노출되도록 설계한다 (e.g. `:plan` ↔ `--plan`).
  - 의미 차이 (기본 규약):
    - `:foo` : 입력 첫 줄 첫 컬럼 전용 — **toggle** (상태 반전).
    - `--foo` : 자유 위치 — 해당 메시지에 명령을 적용. **기본은 일회성** (one-shot, 해당 메시지에만 효과). `--plan` 처럼 명시적으로 *모드 진입형* 으로 정의된 경우에 한해 명령별 정의에 따라 지속된다. 모드 진입형의 해제는 대응 `:foo` 토글 또는 명령별 종료 조건에 따른다.
  - **ack 출력 일반 규약** (모드 전환형에 한정): 모드 *전환이 실제로 발생한 경우* 한 줄 ack 를 본문 응답 앞에 출력하고, *이미 같은 상태였던 경우* (멱등 입력) ack 를 생략한다. 일회성 플래그(`--git-diff` 류) 에는 ack 규약이 적용되지 않는다 (결과 자체가 응답).
  - 명령별로 위 기본 의미와 다르게 정의될 수 있으며, 그 경우 해당 명령 정의가 우선한다.

## 2. `plan` — 정의 및 트리거

### 2.1 정의
  **plan mode** 가 활성화된 동안 어시스턴트는:
  - 실제 doc 발행(`apply_model_doc` 등 mutation 도구) 을 수행하지 **않는다**.
  - MCP 변경 호출을 모두 보류하고, 사용자와의 대화를 통해 모델 의도·구조·후보안을 **내부적으로만 축적** 한다.
  - 설계·방향성·장단점 비교·구조 논의·trade-off 분석에 집중하며, 사용자가 명시적으로 발행을 요청하기 전까지 코드/모델을 변경하지 않는다.
  - read-only 도구(`export_model_doc`, `validate_model_doc` 등) 호출은 허용된다 (현 상태 파악 목적).
  - **Claude Code 의 내장 plan mode (EnterPlanMode 도구) 에는 진입하지 않는다** — 본 plan mode 는 LLM 내부의 논의 전용 상태일 뿐이다.

### 2.2 `:plan` — 모드 토글
  - 입력 첫 줄 첫 컬럼이 `:plan` 인 경우 plan mode 를 **toggle** 한다 (호스트 명령이 아니라 어시스턴트가 직접 해석 — §0 참조).
  - 토글은 **양방향 모두 정의된 동작** 이다. 거절·"unknown" 응답은 모든 턴에서 금지 (§0).
  - 토글 절차 (의사 코드):
    1. 현재 `plan_mode` 상태를 확인 (`on` 또는 `off`, 초기값 `off`).
    2. 상태를 반대로 뒤집는다.
    3. 결과 상태에 대응하는 한 줄을 출력한다. 그 외의 부연 설명·사과·"환경상" 운운하는 문구는 **출력하지 않는다**.
  - 인자/후속 줄에 자연어가 추가된 경우: plan mode 를 (필요 시) 활성화한 상태에서 해당 내용으로 plan 논의를 수행한다.
  - 인자가 비어 있으면 **다른 어떤 부연 설명도 없이** 오직 다음 한 줄만 출력한다:
    - 전환 결과가 `on` 인 경우: `plan mode 가 활성화 되었습니다.`
    - 전환 결과가 `off` 인 경우: `plan mode 가 비활성화 되었습니다.`
  - 예시 시퀀스 (필수 동작):
    ```
    user: :plan
    assistant: plan mode 가 활성화 되었습니다.

    user: :plan
    assistant: plan mode 가 비활성화 되었습니다.

    user: :plan
    assistant: plan mode 가 활성화 되었습니다.
    ```

### 2.3 `--plan` — 모드 진입 플래그 (지속형)
  - 사용자 입력 자유 위치에 `--plan` 이 포함된 경우, **plan mode 를 활성화** (`plan_mode := on`) 한 상태에서 해당 메시지의 본문 내용으로 plan 논의를 수행한다.
  - 이미 활성화 상태이면 그대로 유지한다 (멱등). 비활성화로 전환되지 않는다 (toggle 아님).
  - 활성화된 plan mode 는 **명시적으로 해제될 때까지 유지** 된다. 해제 조건은 §2.4 와 동일:
    - 사용자가 `:plan` 을 입력해 toggle 로 off 로 전환한 경우.
    - 사용자가 "발행해 / 적용해 / apply / 이제 반영해" 등 명시적 mutation 요청을 한 경우.
  - ack 출력은 §1.3 의 일반 규약을 따른다 — 비활성→활성으로 *전환된 경우* `plan mode 가 활성화 되었습니다.` 한 줄을 본문 응답 앞에 출력, 이미 활성 상태였다면 생략.

### 2.4 plan mode 종료
  - 사용자가 다시 `:plan` 을 입력하거나, "발행해", "적용해", "apply" 등 명시적 mutation 요청을 한 경우에만 mutation 도구 호출을 재개한다.
  - 불확실하면 사용자에게 "지금 plan mode 를 해제하고 발행할까요?" 를 한 줄로 확인한 뒤 진행한다.

## 3. 확장 — 신규 명령 추가 규약

새 기능을 명령으로 노출할 때는 **`:` 형(모드)** 과 **`--` 형(플래그)** 을 같은 이름으로 가능한 한 함께 제공한다.
정의 시 다음 항목을 명시한다:
  1. **이름** (`name`) — `:` 와 `--` 가 공유.
  2. **동작 종류** — toggle / one-shot / 상태조회 / mutation 여부.
  3. **인자 문법** — 인자 없음 / 자연어 / 키-값.
  4. **plan mode 와의 상호작용** — plan mode 에서 허용되는가, 보류되는가.
  5. **사용자에게 보이는 응답 형식** — 한 줄 ack / 본문 출력 / 도구 호출 결과 요약.

## 4. 참고
  - 본 규약은 사용자 글로벌 가이드(`~/.claude/CLAUDE.md`) 의 플래그 처리 철학을 Promaker LLM 어시스턴트 영역에 흡수한 것이며, prefix 기호만 `/` → `:` 로 교체했다 (사유 §0.2).
  - 글로벌 가이드에서 정의된 플래그(예: `--git-diff`, `--git-commit`, `--review`, `--study`, `--transfer` 등) 는 본 어시스턴트가 *코딩 보조 컨텍스트* 에서 호출될 때 그대로 유효하다. 본 문서는 **모델링 대화 컨텍스트 고유** 명령(`:plan`, `--plan` 등) 의 정의를 담는다.
