# DS Runtime Mode 상태 타임라인

Work/Call 계층 구조에서 RGFH 상태 전이와 InTag/OutTag 핸드셰이크의 모드별 동작을 정리한 문서.

## 상태 범례

### RGFH 상태 색상

| 상태 | 표기 | 의미 |
|------|------|------|
| **R** (Ready) | 🟢 녹색 | 준비 상태 |
| **G** (Going) | 🟡 노란색 | 진행 중 |
| **F** (Finish) | 🔵 파란색 | 완료 |
| **H** (Homing) | ⚪ 밝은 회색 | 복귀 |

### InTag/OutTag 신호

| 신호 | 표기 | 의미 |
|------|------|------|
| **InTag ON** | 🟦 하늘색 네모 | 입력 신호 활성 |
| **OutTag ON** | 🟪 분홍 네모 | 출력 신호 활성 |
| **OFF** | ⬜ 흰 네모 | 신호 비활성 |

### 계층 구조

```
Work (상위)
 ├── Call1 (하위 1) ── In/Out 핸드셰이크
 └── Call2 (하위 2) ── In/Out 핸드셰이크
```

---

## 1. Simulation / Control 모드

시스템이 직접 제어를 주도하는 모드. Out 신호를 시스템이 발행하고, In 신호로 완료를 확인한다.

### 타임라인

| 시간 | Work1 | Call1.In / Out (Status) | Call2.In / Out (Status) | 비고 |
|------|-------|------------------------|------------------------|------|
| t0 | ⚪H | ⬜ / ⬜ (⚪H) | ⬜ / ⬜ (⚪H) | 초기 부팅: Homing |
| t0.5 | ⚪H | ⬜ / ⬜ (⚪H) | 🟦 / ⬜ (⚪H) | 원위치 시작 (Call2 동작대상) |
| t0.7 | ⚪H | ⬜ / ⬜ (⚪H) | 🟦 / ⬜ (⚪H) | 원위치 센서 감지 |
| t0.9 | ⚪H | ⬜ / ⬜ (⚪H) | ⬜ / ⬜ (⚪H) | 원위치 복귀: Out=1->0, In=1 유지 |
| t1 | 🟢R | ⬜ / ⬜ (🟢R) | ⬜ / ⬜ (🟢R) | ⚪H->🟢R 전환 (원위치 완료) |
| t2 | 🟡G | ⬜ / ⬜ (🟢R) | ⬜ / ⬜ (🟢R) | Work 🟡G=1 |
| t3 | 🟡G | ⬜ / 🟪 (🟢R) | ⬜ / ⬜ (🟢R) | Call1 🟡G=1 -> Call1.Out=1 |
| t3.5 | 🟡G | ⬜ / 🟪 (🟡G) | ⬜ / ⬜ (🟢R) | 상호 리셋: Call1.Out ON -> Call2.In OFF |
| t4 | 🟡G | ⬜ / 🟪 (🔵F) | ⬜ / ⬜ (🟢R) | Call1 완료 -> Call1.In=1 |
| t4.5 | 🟡G | 🟦 / ⬜ (🔵F) | ⬜ / ⬜ (🟢R) | In 후 Out OFF: Call1.In ON -> Call1.Out OFF |
| t5 | 🟡G | ⬜ / ⬜ (🔵F) | ⬜ / 🟪 (🟡G) | Call2 🟡G=1 -> Call2.Out=1 |
| t5.5 | 🟡G | ⬜ / ⬜ (🔵F) | ⬜ / 🟪 (🟡G) | 상호 리셋: Call2.Out ON -> Call1.In OFF |
| t6 | 🟡G | ⬜ / ⬜ (🔵F) | ⬜ / 🟪 (🔵F) | Call2 완료 -> Call2.In=1 |
| t6.5 | 🟡G | ⬜ / ⬜ (🔵F) | 🟦 / ⬜ (🔵F) | In 후 Out OFF: Call2.In ON -> Call2.Out OFF |
| t7 | 🔵F | ⬜ / ⬜ (🔵F) | ⬜ / ⬜ (🔵F) | 모든 Call 🔵F -> Work 🔵F |
| t8 | ⚪H | ⬜ / ⬜ (⚪H) | ⬜ / ⬜ (⚪H) | Work ⚪H -> Call ⚪H, Call2.In=1 유지 |
| t9 | 🟢R | ⬜ / ⬜ (🟢R) | ⬜ / ⬜ (🟢R) | 다음 사이클 준비 (t1과 동일 상태) |

### 동작 단계 요약

| 구간 | 동작 | 설명 |
|------|------|------|
| t0 ~ t1 | Homing -> Ready | 초기 부팅 후 원위치 복귀 완료 |
| t2 | Work -> G | Work가 먼저 Going 전환 |
| t3 ~ t4.5 | Call1 실행 사이클 | G=1 -> Out=1 -> 완료 -> In=1 -> Out OFF |
| t5 ~ t6.5 | Call2 실행 사이클 | G=1 -> Out=1 -> 완료 -> In=1 -> Out OFF |
| t7 | 모든 Call F -> Work F | 하위 완료 -> 상위 완료 (상향 전파) |
| t8 ~ t9 | H -> R | 사이클 리셋, 다음 사이클 준비 |

### Sim/Control 모드 핵심 특성

1. **순차 실행**: Call1 -> Call2 순서로 실행 (병렬 아님)
2. **상호 리셋 메커니즘**: Call의 OutTag ON 시 -> 상대 Call의 InTag OFF로 리셋
3. **Out -> In 핸드셰이크**: Call 완료 후 In=1 수신 -> 그 후에야 Out OFF (확인 후 해제)
4. **상향 전파**: 모든 하위 Call이 F가 되어야 Work가 F로 전환
5. **제어 주체가 시스템**: 시스템이 G=1, Out=1 등을 직접 발행

## 2. Monitoring / VirtualPlant 모드

외부(가상 플랜트)가 신호를 주도하는 모드. 시스템은 외부 In 신호를 수신하고 반응한다.

### 타임라인

| 시간 | Work1 | Call1.In / Out (Status) | Call2.In / Out (Status) |
|------|-------|------------------------|------------------------|
| t0 | ⚪H | 🟦 / ⬜ (⚪H) | ⬜ / ⬜ (⚪H) |
| t0.5 | ⚪H | ⬜ / ⬜ (⚪H) | ⬜ / ⬜ (⚪H) |
| t1 | 🟢R | 🟦 / ⬜ (🟢R) | ⬜ / ⬜ (🟢R) |
| t1.5 | 🟡G | ⬜ / ⬜ (🟢R) | ⬜ / ⬜ (🟡G) |
| t2 | 🟡G | ⬜ / ⬜ (🟢R) | ⬜ / ⬜ (🟢R) |
| t3 | 🟡G | ⬜ / ⬜ (🟡G) | ⬜ / ⬜ (🟡G) |
| t3.5 | 🟡G | ⬜ / ⬜ (🟡G) | ⬜ / ⬜ (🟡G) |
| t4 | 🟡G | ⬜ / ⬜ (⚪H) | ⬜ / ⬜ (🔵F) |
| t5 | 🟡G | ⬜ / ⬜ (🟡G) | ⬜ / ⬜ (🟡G) |
| t5.5 | 🟡G | ⬜ / ⬜ (🟡G) | ⬜ / ⬜ (🟡G) |
| t6 | 🟡G | ⬜ / ⬜ (🔵F) | ⬜ / ⬜ (🔵F) |
| t6.5 | 🟡G | ⬜ / ⬜ (🔵F) | ⬜ / ⬜ (🔵F) |
| t7 | 🔵F | ⬜ / ⬜ (🔵F) | ⬜ / ⬜ (🔵F) |
| t8 | ⚪H | ⬜ / ⬜ (⚪H) | ⬜ / ⬜ (⚪H) |
| t9 | 🟢R | ⬜ / ⬜ (🟢R) | ⬜ / ⬜ (🟢R) |
| t10 | 🟡G | ⬜ / ⬜ (🟢R) | ⬜ / ⬜ (🟡G) |

### Monitor/VirtualPlant 모드 핵심 특성

1. **외부 신호 기반**: InTag가 외부(VirtualPlant)에서 들어오므로, 시스템은 관찰/응답 역할
2. **초기부터 In 활성**: t0에서 이미 Call1.In이 ON -- 외부 플랜트가 먼저 신호를 보낸 상태
3. **사이클이 1단계 더 김**: t10까지 존재하여 다음 사이클 진입까지 포함

## 모드 비교

### Sim/Control vs Monitor/VirtualPlant 차이점

| 항목 | Sim/Control | Monitor/VirtualPlant |
|------|-------------|----------------------|
| **t0 초기 InTag** | 모두 OFF | Call1.In = ON |
| **신호 주도** | 시스템이 Out 발행 | 외부에서 In 신호 수신 |
| **타이밍** | t0.5~t0.9 Homing 세분화 | t0->t1 직행 (R 전환 빠름) |
| **사이클 끝** | t9에서 종료 | t10 존재 (다음 사이클 시작) |
| **제어 방향** | Push (시스템 -> 외부) | Pull/React (외부 -> 시스템) |

### 공통 원칙

- **RGFH 상태 전이 순서**: H -> R -> G -> (Call 실행) -> F -> H -> R (반복)
- **상향 전파**: 하위(Call) 전체 완료 -> 상위(Work) 완료
- **In/Out 핸드셰이크**: 동기화를 위한 신호 교환
- **상호 리셋**: 다음 Call 실행 전 이전 신호 정리

### 설계 요점

1. **Sim/Control은 Push 모델** -- 시스템이 Out으로 명령을 밀어넣고, In으로 완료 확인
2. **Monitor/VirtualPlant는 Pull/React 모델** -- 외부 In 신호에 반응하여 상태 전이
3. **상호 리셋**이 양쪽 모두 존재하여, 다음 Call 실행 전 이전 신호를 깨끗이 정리
4. 이 구조는 IEC 61131 스타일의 프로그램-태스크 계층과 유사한 산업 자동화 패턴
