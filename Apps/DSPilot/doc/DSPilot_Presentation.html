Why

왜 Projection 중심 구조로 다시 묶는가

현재 구조의 문제

  • UI에서 데이터 조합과 계산이 섞이기 쉽다.
  • 이벤트 처리, 통계 계산, Flow 집계가 여러 서비스로 흩어져 있다.
  • 좁은 update 메서드만으로는 상세 projection을 안정적으로 유지하기 어렵다.
  • `DROP TABLE` 방식은 누적 통계와 화면 신뢰성을 무너뜨린다.
  • `WorkName` 정확도 같은 메타데이터 오류가 전체 화면 품질을 무너뜨린다.

문서가 제시하는 방향

  • UI는 `dspFlow`, `dspCall`만 읽는다.
  • 계산은 Engine에서 완료된 projection으로 제공한다.
  • 업데이트는 patch 기반으로 넓게 처리한다.
  • AASX 구조와 PLC 이벤트를 하나의 projection 흐름으로 묶는다.
  • 기능 구현은 projection 필드 정의에서 시작한다.

아키텍처 기준 문장
AASX는 구조를 제공하고, PLC 이벤트는 상태를 제공하며, Engine은 이 둘을 합쳐서 `화면이 바로 읽을 수 있는 Projection`으로 유지한다.
Projection Pattern

세 가지 계산 책임

1. Static Bootstrap

  • 트리거: 프로젝트/AASX 로드 시
  • 역할: Flow, Work, Call 구조 적재
  • 계산: topology, sequence, tag mapping, device
  • 출력: `dspFlow`, `dspCall` 초기 projection

2. Runtime Event Update

  • 트리거: PLC value change
  • 역할: Rising Edge 감지, Tag → Call 매핑
  • 계산: state transition, start/finish timestamp
  • 출력: call state, active count, cycle start/end

3. Aggregate Recompute

  • 트리거: Call 완료, Tail 완료
  • 역할: 통계와 Flow 집계 계산
  • 계산: Avg, StdDev, Min, Max, MT, WT, CT
  • 출력: `FocusScore`, `SlowFlag`, Flow KPI
계산 책임 주요 입력 핵심 산출물
Static Bootstrap AASX System / Flow / Work / Call WorkName, SequenceNo, Prev, Next, IsHead, IsTail, InTag, OutTag
Runtime Event Update PLC tag rising edge State, LastStartAt, LastFinishAt, LastTransitionAt, ActiveCallCount
Aggregate Recompute Call 완료 결과 AverageGoingTime, StdDevGoingTime, MT, WT, CT, FocusScore
Database Schema

Projection 테이블 구조

테이블 역할 대표 필드 업데이트 시점
dspFlow Flow 레벨 projection State, MovingStartName, MovingEndName, ActiveCallCount, ErrorCallCount, MT, WT, CT, LastCycleStartAt, LastCycleEndAt 프로젝트 로드, call 상태 변경, tail 완료
dspCall Call 레벨 projection WorkName, SequenceNo, State, ProgressRate, LastStartAt, LastFinishAt, LastDurationMs, AverageGoingTime, StdDevGoingTime, FocusScore 프로젝트 로드, In/Out tag 이벤트, 오류 이벤트
plcTagLog 원시 PLC 이력 TagName, Value, Timestamp, Source PLC 수집 시

중요 규칙

  • `dspFlow`, `dspCall`은 화면용 read model이다.
  • UPSERT 시 기존 통계는 보존한다.
  • 누적 필드와 정적 메타 필드는 구분해서 업데이트한다.

선행 수정 포인트

  • `WorkName = flow.Name` 버그 수정
  • `DROP TABLE` 제거 후 migration 전환
  • 좁은 update 메서드를 patch 메서드로 교체

핵심 메시지

  • 기능 구현은 화면부터 시작하지 않는다.
  • 먼저 필요한 projection 필드를 정의한다.
  • 그 다음 update trigger를 정의한다.
  • 마지막에 UI는 read-only로 붙인다.

UI 금지사항

  • Mapper 재호출
  • Raw log 직접 스캔으로 평균/표준편차 재계산
  • Head/Tail 재판정
  • Focus score를 UI에서 재계산
Refactoring Plan

리팩토링 우선순위

Phase 1

데이터베이스 스키마 확장

`dspFlow`, `dspCall`에 projection 필드를 추가하고 migration 기반 관리로 전환한다.

Phase 2

Repository Patch 메서드 구현

좁은 update 메서드를 `FlowPatch`, `CallPatch` 기반 업데이트로 교체한다.

Phase 3

WorkName 버그 수정

`flow.Name`이 아니라 `work.Name`을 저장해 Work 단위 화면과 통계를 바로잡는다.

Phase 4

Bootstrap 모듈 재설계

정적 메타데이터와 누적 통계를 분리하고, UPSERT 시 통계를 보존한다.

Phase 5-7

Runtime / Statistics / Flow Aggregation 재정비

이벤트 처리, 통계 계산, Flow 집계를 patch 기반 projection writer로 일관되게 묶는다.

Phase 8

C# Service Layer 단순화

UI/서비스 계층은 projection 읽기에 집중하고, 계산 책임을 Engine으로 되돌린다.

Invariants

반드시 지켜야 하는 불변 규칙

UI는 계산 금지

화면은 `dspFlow`, `dspCall`만 읽는다. 평균, 표준편차, FocusScore를 다시 계산하지 않는다.

Projection은 완전해야 함

화면이 필요한 핵심 값은 projection에 직접 존재해야 한다. 화면에서 join/recompute 하지 않는다.

DROP TABLE 금지

통계 보존이 핵심이다. 스키마 변경은 migration과 UPSERT로 처리한다.

WorkName 정확도

Call은 반드시 `work.Name`을 가진다. 이 값이 틀리면 Heatmap, Flow Detail, 병목 분석이 전부 흔들린다.

Patch Update

한 서비스가 필요한 필드만 patch로 갱신한다. 전체 row를 무분별하게 덮어쓰지 않는다.

Raw Log는 보조 역할

`plcTagLog`는 디버그와 검증용이다. 기본 UI는 projection 중심으로 설계한다.

Outcome

기대 효과와 검증 기준

기대 효과

  • UI 단순화: 화면은 projection 읽기에만 집중
  • 성능 향상: 실시간 계산을 UI에서 제거
  • 일관성 향상: Flow/Call 상태와 통계가 하나의 모델로 정리
  • 확장성 확보: 새 기능 추가 시 projection 필드만 확장
  • 운영 안정성: DROP TABLE 제거로 누적 통계 보존

검증 기준

  • 프로젝트 로드 후 `WorkName`, topology, tag mapping 정확성 확인
  • PLC 이벤트 수신 후 state / timestamp / duration이 projection에 즉시 반영되는지 확인
  • Tail 완료 시 MT / WT / CT가 정확히 계산되는지 확인
  • UI가 raw log 없이 필요한 화면을 그릴 수 있는지 확인
  • migration 후 기존 통계가 유지되는지 확인
최종 결론
DSPilot의 핵심은 화면을 많이 만드는 것이 아니라, `AASX 구조 + PLC 이벤트 + 통계 계산`을 일관된 projection으로 유지하는 엔진을 만드는 것이다. 발표자료도 그 구조를 기준으로 이해시키는 방향으로 정리해야 한다.
